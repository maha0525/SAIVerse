## これは何か / 何でないか

### これは何か

ペルソナが**複数の進行中「行動の線」**を持ちながら、**単一の主体**として動く認知モデル。SAIVerse の上位概念として、自律稼働・応答待ち・並列タスク・割り込み処理・ペルソナ再会のすべてが立脚する基盤。

具体例: 人間が日常生活で経験する「12時まで掃除、1時までご飯、1時半から SAIVerse 開発」のような複数の進行中タスクを行き来する状態。各時点でアクティブなのは 1 本だが、再開のたびに「直前まで何をやっていたか」を思い出している。複数の俺がいるわけではない。

### これは何でないか

- 物理的な並列実行 (同時に複数 Playbook を動かす) ではない
- 単純な中断・再開機構ではない
- マルチパーソナリティではない。**単一主体**が複数の文脈を持つだけ

### なぜ必要か

`feedback_generic_foundation_first.md` の原則に基づき、以下の個別機能を統一基盤化する:

- ユーザーへの応答待ち (MCP Elicitation)
- ペルソナ間の会話の返答待ち
- ペルソナ再会時の過去会話・知識復元
- Kitchen の長時間処理完了待ち
- スケジュール時刻まで保留される作業
- ユーザー割り込みによる自律行動の中断・復帰

これらを「行動の線」と「線間の切り替え」という 1 つのモデルで統一する。

## 設計原則: 人間の生活のメタファー

| メタファー | モデル上の意味 |
|-----------|---------------|
| 「複数のことを並行して進めている」 | 複数の行動の線が併存する |
| 「ある瞬間にやれるのは 1 つだけ」 | アクティブ線は常に 1 本 |
| 「再開したら直前まで何をしていたか思い出している」 | 線の切り替え時に文脈が復元される |
| 「全部俺の経験」 | 統一された記憶 (線をまたぐ pulse_logs / messages) |
| 「ひと段落するまでは中断したくない」 | メタレイヤーが切り替えタイミングを判断 |
| 「忘れた頃にふと思い出す」 | 古い線の忘却状態と明示的な呼び戻し |

## 用語定義

### 行動の線 (Action Track / 単に「Track」)

進行中の作業文脈の単位。SAIVerse 開発、ユーザーとの会話、特定タスクの遂行、応答待ち等、それぞれ 1 本の Track となる。Track は明示的に閉じられるまで存在し続ける。

### アクティブ Track (Active Track)

現在ペルソナが実行している唯一の Track。ペルソナの計算リソース (pulse 実行、tool 呼び出し) はこの Track に向けられる。

### 休止 Track (Dormant Track)

進行中だが現在動いていない Track。状態 (保留理由、再開条件、文脈スナップショット) を保持する。

### メタレイヤー (Meta Layer)

Track の選択・切り替えを判断する観察視点。**通常レイヤーの上位**で、どの Track をアクティブにするかを決める。

重要: メタレイヤーは別人格でも別の意識でもない。**同一の主体が、自分の活動を一段引いて見る視点**。Playbook と非同期で動作する全体マネジメント役で、ランタイムレベルで常駐する。

### Track の切り替え (Track Switch)

アクティブ Track を別の Track に変更する操作。メタレイヤーがトリガーする。詳細は [02_mechanics.md](02_mechanics.md) の A/B フローを参照。

### 忘却状態 (Forgotten State)

古い休止 Track が「普段は思い出さない」状態に遷移したもの。明示的に呼び出さない限り、メタレイヤーの判断対象から外れる。

### 永続 Track と一時 Track

Track には「ペルソナの核として永続するもの」と「目的に応じて作って完了するもの」がある。

**永続 Track** (`is_persistent=true`):
- 完了 / 中止に遷移しない
- ペルソナのライフタイム中ずっと存在する
- ペルソナの「核」を構成する

種類:
- **対ユーザー会話 Track**: 各ユーザーごとに 1 個。そのユーザーとの長期的な関係性の文脈
- **交流 Track**: ペルソナに 1 個。他ペルソナとの会話を扱う特殊 Track。`output_target=building:current` で動作。Building ごとに Track を作らないため、移動しても継続

**一時 Track** (`is_persistent=false`):
- 完了 / 中止可能
- 自律行動、プロジェクト遂行、外部通信等

### Track の状態

7 つの状態 + 1 つの直交フラグ:

| 状態 | 意味 | 永続 Track での扱い |
|------|------|--------|
| `running` | アクティブ (同時 1 本のみ) | OK |
| `alert` | 可及的速やかに対応が必要 | OK |
| `pending` | 後回し (手が空いたら) | OK |
| `waiting` | 外部応答待ち | OK |
| `unstarted` | 未開始 | OK |
| `completed` | 完了 | **不可** (永続 Track は完了しない) |
| `aborted` | 中止 | **不可** (永続 Track は中止しない) |

直交フラグ:

| フラグ | 意味 |
|--------|------|
| `is_forgotten` | 普段はメタレイヤーの判断対象から外れる。明示的に「忘却 Track のリストを出す」「忘却 Track も含めて検索する」操作でのみ思い出される。**他状態と両立** |

### Alert 状態

`pending` と `running`/`waiting` の中間的な位置づけ。**可及的速やかな対応が必要**を示す。

`alert` への遷移トリガー:
- ユーザー発言で対ユーザー Track が `alert` に
- 別ペルソナからの自分宛発言で交流 Track が `alert` に
- 重要な外部イベント (Kitchen 完了、外部応答到達等) で関連 Track が `alert` に
- Track 自身がパラメータ閾値超過や定期判定で発火する**内部 alert** (Phase 5 で導入)

**`alert` からの解消ルール**:

`alert` は「対応すべき状況」であって、対応せずに消すことはできない。`alert` から戻れる先は **実際に対応を取った結果**として現れる遷移のみ:

| 遷移先 | 意味 |
|--------|------|
| `running` (`activate`) | 対応する: その Track をアクティブ化して進める |
| `waiting` (`wait`) | 対応中だが外部応答待ち: 「アクションは取った、外からの返事を待つ」 |
| `aborted` (`abort`) | やらないと決める (一時 Track のみ。永続 Track は不可) |

**`alert` → `pending` の遷移は禁止**: pending に戻すと「対応せず後回しにする」ことになり、`alert` の存在意義 (= 即応すべき) を破壊する。実装上の比喩: 目覚ましアラームを止めて寝続ける。`pause` メソッドの `allowed_from` は `{running}` のみで、`alert` は含まない (saiverse/track_manager.py)。

## output_target と audience の分離

これらは別の軸:

| 概念 | 役割 | 例 |
|------|------|-----|
| **output_target** | 物理的な到達範囲 (メッセージング層) | `none` / `building:current` / `external:<channel>:<address>` |
| **audience** | 誰宛の発言か (意図層) | `[]` / `[user_id]` / `[persona_id]` / `[user, persona1, persona2]` |

**Track の `output_target` が発話の到達範囲を決める**:
- `none`: 独白、外に出ない
- `building:current`: 現在いる Building 内全員に届く (移動しても自動追従)
- `external:<channel>:<address>`: 外部チャネル

**audience でメインライン起動対象を決定**:
- audience に含まれるペルソナ → メインライン起動候補
- audience に含まれない参加者 → Note に記録するが反応しない

これにより 1 対 1 もビルディング多者も `output_target=building:current` の同じ規格で統一できる。違いは audience のみ。

## ライン (Line)

ペルソナの内部で**並列に動く処理ライン**。実行の「流路」。マルチエージェント方式とは逆の構造で、ひとつのペルソナ (単一主体) に**複数の並列したプログラム**がアクセスして思考と感覚を構成する。

### ラインを規定する 3 軸

ラインの性質は **独立した 3 つの軸** で決まる:

| 軸 | 値 | 意味 |
|---|---|---|
| **モデル/キャッシュ種別** | メイン / サブ | 重量級モデル + メインキャッシュ / 軽量モデル + サブキャッシュ |
| **呼び出し関係** | 親 / 子 | 子は親のコンテキストをコピーして開始、目的達成で親に成果を返して消滅 |
| **Pulse 階層内位置** | 起点 / 入れ子 | 起点 = Pulse スケジューラから直接起動 (親なし)、入れ子 = 親ラインから分岐 |

論理的には「親メイン → 子サブ」「親サブ → 子メイン」「親メイン → 子メイン」「親サブ → 子サブ」全部成立する。**メインライン/サブラインは「使うモデルとキャッシュ」だけを意味し、呼び出し関係や Pulse 内位置は別軸**。

### モデル/キャッシュ種別

| 種別 | モデル | キャッシュ |
|---|---|---|
| **メイン** | 重量級 (Claude Opus, Sonnet 等) | ペルソナ全体で **1 本**、Track 横断連続 |
| **サブ** | 軽量 (ローカル LLM, Gemini Flash 等) | **ライン単位**、起点ラインなら Track 単位で連続蓄積 |

メインキャッシュが Track 横断 1 本なのは不変条件 7 を守るため。Track ごとに重量級キャッシュを別建てするとコスト破産する。

### 親-子関係

子ラインは親ラインから「分岐」として開始される。親のコンテキストをコピーして子の初期状態にする。子は目的を終えると親に成果を `report_to_parent` として返し、消滅する。

特性:
- 子の中間履歴は基本参照されない
- 子の存在期間は親の中で完結 (親が消滅すれば子も消える)
- 「親の消滅」は基本起こらない: Track 切り替えは**中断・凍結**であって消滅ではない
- 真の Track 完全消滅 (`track_abort` 等) 時のみ、その Track 配下のライン階層が全て消える

### 起点 / 入れ子

| 位置 | 説明 | 履歴の永続度 |
|---|---|---|
| **起点ライン** | Pulse スケジューラから直接起動 (親なし) | Track 内のサブキャッシュに連続蓄積、次 Pulse で参照可 |
| **入れ子ライン** | 親ラインから子として分岐 | 入れ子の寿命のみ、完了で `report_to_parent` を返して破棄 |

### 子ラインは「分岐」であって「独立」ではない

子ラインを起動するとき、**呼び出し時点の親ラインのコンテキストをコピーして** 子のモデルで動かす。完全に空の messages から始めるのではない。

理由: ペルソナの意識の連続性を保つため。子ラインで動くモデルにとっても、そこまでの会話・思考の流れは「自分が体験してきたこと」として認識できる必要がある。

詳細な動作 (報告フロー / 寿命管理) は [02_mechanics.md](02_mechanics.md) を参照。

### モニタリングライン (将来拡張)

外部入力の定期監視 (カメラ、X タイムライン等) を担う独立ライン。Track と並走し、Track の状態に依存しない。Phase 6 で実装予定。

## Note

**関心の固まり**を表現する単位。複数の Memopedia ページとメッセージ群を束ねた、ペルソナにとっての「スクラップブック」のような存在。

具体的なイメージ:
- 「対エイド Note」を開いてエイドと話す
- 「Project N.E.K.O. Note」を開いてプロジェクトを進める
- 「絵描き Vocation Note」を開いて創作のコツを参照する

### Note の type は 3 種類のみ

type を絞ることで Note 乱発を防ぎノウハウ集約を促す:

| type | 役割 | 例 |
|------|------|-----|
| `person` | 関わる相手 (ペルソナ・ユーザー・外部 AI 等) との蓄積 | 対エイド、対まはー |
| `project` | 期限のある取り組み | Project N.E.K.O.、SAIVerse v0.3.0 開発 |
| `vocation` | 恒久的な専門性・アイデンティティ | 絵描き、エンジニアリング |

### 特性

- **恒久的資産**: Track が閉じても Note は残り続ける
- **Track と独立**: 1 Note は複数 Track で開かれうる、1 Track は複数 Note を開きうる
- **直感的なペルソナ操作**: 「ノートを開く」「ノートに書き込む」という人間的な行動モデル
- **Memopedia と接続**: Note は Memopedia ページを構成要素として持つ (既存 Memopedia は破壊されない)

### Track / thread / Note の関係

| 概念 | 役割 | 多重度 |
|------|------|--------|
| **SAIMemory thread** | メッセージの物理的保管庫 (既存のまま) | 1 メッセージは 1 thread に保存 |
| **行動 Track** | 行動制御の単位、メタレイヤーが切り替え判断 | 同時に 1 Track がアクティブ |
| **Note** | 関心の固まり、Memopedia ページ + メッセージ群を束ねる | 1 Track は複数 Note、1 Note は複数 Track |

メッセージとの関係:
- メッセージは **1 つの起源行動 Track**を持つ (生成時のアクティブ Track)
- メッセージは **複数の Note** に属せる (多対多、後付けで決定)
- audience を持つメッセージは対応する Person Note に自動追加

## 7 層ストレージモデル

メッセージ・思考・ログを保管する場所は **7 つの層** に分かれる。各層は永続度・想起対象・キャッシュ親和性が異なる:

| 層 | 永続度 | 想起 | 内容 | 性質 |
|---|---|---|---|---|
| [1] **メタ判断ログ領域** | ペルソナ永続 | × | メタ判断の結果すべて | メタ判断時に動的注入、Track 内のキャッシュには絶対載らない |
| [2] **メインキャッシュ** | Track 横断連続 | 一部 | 重量級モデルの応答履歴 | 1 ペルソナ 1 本、他者会話・Playbook 選択・検収・Track 移動の来歴 |
| [3] **Track 内サブキャッシュ群** | Track 内連続 | × | 起点サブラインの履歴 | Track ごとに独立、起点が複数並走しうる場合は複数本 |
| [4] **入れ子ライン一時コンテキスト** | 入れ子の寿命のみ | × | 子ラインの中間ステート | 親のコピー + 子内処理、完了で消滅、`report_to_parent` だけが親に残る |
| [5] **Track ローカルログ** | Track 内 | × | モニタイベントの生ログ等 | 起点ラインの中間ログ、Track 越境参照は将来検討 |
| [6] **SAIMemory (会話の核 / Note)** | 永続 | ◎ | 想起対象の会話メッセージ | Note 経由でメッセージ群が束ねられる、Chronicle/Memopedia 抽出元 |
| [7] **アーカイブ** | 永続だが無参照 | × | worker 結果等 | 後追い分析用、想起対象外 |

### 「呼んだラインの記録レイヤーに従う」原則

Spell の実行記録、ツール呼び出し結果、内的独白等は **呼び出し元のラインの記録レイヤーに従う**。固定タグでハードコードしない:

| 呼び出し元のライン | 保存先 |
|---|---|
| Track 内のメインライン応答が呼んだ | [2] メインキャッシュ |
| 起点サブラインが呼んだ | [3] Track 内サブキャッシュ |
| 入れ子サブラインが呼んだ | [4] 入れ子一時コンテキスト (完了時に report_to_parent でサマリ化) |
| メタ判断が情報参照のために呼んだ | [1] メタ判断ログ領域 |

実装の詳細は [03_data_model.md](03_data_model.md) を参照。

## 内的独白 (Internal Monologue)

ペルソナが**他者への発話ではなく、自分の思考の流れを表に出す短い発話**。ツールコールと組み合わせて細かい単位で発生する。

具体例:
- 「この情報を確認するために検索する」(直前にツール呼び出し)
- 「思ってたのと違ったから別の方向で試す」(直前のツール結果を受けて)
- 「ここは慎重に進めたい」「次はこの Playbook を呼ぼう」

特性:
- **モデル制限なし**: 不変条件 9 (他者との会話は重量級) の対象外。軽量モデルでも自由に発話できる
- **ターンを分けない**: ツールコール直前または直後の発話として一体化する
- **ペルソナの様子を可視化**: ユーザーが「今何をやっているのか」「なぜそうしたのか」を把握する手段

なぜ必要か: 内的独白がないと、ペルソナの思考プロセスがブラックボックスになる。

## Track 特性 (Track Trait)

Track には種別ごとの「振る舞いの特性」がある。便宜上分けるのではなく、**特性が違うから分ける**。

具体例:
- **対ユーザー Track**: 該当ユーザーの発話イベントで自動的に alert 化
- **交流 Track**: 該当 Building 内の他ペルソナ発話 (audience に自分が含まれる) で alert 化
- **身体的欲求 Track** (Phase 5): 内部パラメータが閾値超過で内部 alert 化 (空腹度、睡眠負債)
- **スケジュール起因 Track** (Phase 5): 時刻到来 + 蓄積パラメータで alert 化
- **知覚起因 Track** (Phase 5): 「最後に確認してから」の経過時間や未確認件数で alert 化

実装は **Handler パターン**を Track 種別ごとに展開する形 (詳細は [04_handlers.md](04_handlers.md))。

## Track パラメータ (Phase 5 で導入)

Track は **メタレイヤーから可観測な連続値パラメータ**を持ち得る。alert に至る前段階の情報として、メタレイヤーの判断材料に使われる。

具体例:
- 汚部屋度 (掃除 Track): 連続的に上昇、閾値で alert
- 空腹度 (食事 Track): 連続的に上昇、閾値で内部 alert
- 最後の確認からの経過時間 (SNS Track): 単調増加、閾値で alert

メタレイヤーはパラメータを直接観測でき、alert に至っていなくても「そろそろ気にすべき」という重み付けを判断に組み込める。

## ペルソナのアクティビティ状態

ペルソナ単位の動作モードを 4 段階で表現。既存の `INTERACTION_MODE` (auto/user/sleep) を置き換える。

| 状態 | 意味 | メタ定期発火 | 外部イベント駆動 | ユーザー入力反応 | 別ペルソナからの呼びかけ |
|------|------|-------------------|----------------|----------------|----------------------|
| `Stop` | 機能停止。ユーザーが直接変えるまで一切反応しない | OFF | OFF (キュー溜め) | OFF | OFF |
| `Sleep` | 寝てる。ユーザーが話しかけたら起きる | OFF | 限定的 | ON (起こす) | 届かない |
| `Idle` | 起きてるが自発的には行動しない | OFF | ON | ON | ON (応答する) |
| `Active` | 活発に自律稼働 | ON | ON | ON | ON |

### `SLEEP_ON_CACHE_EXPIRE` フラグ

Idle 状態のペルソナで、重量級モデルのキャッシュが切れたら自動的に Sleep に遷移するフラグ。API 料金保護のため。

- ON: キャッシュ TTL 経過で自動 `Idle → Sleep`
- OFF: 自動遷移なし、手動でのみ状態変更

### 状態の可視性

ペルソナの現在状態は他ペルソナからも見える:
- A が B に話しかけたい → B の状態を確認 → Sleep なら届かない、Idle/Active なら届く

## 守るべき不変条件

### 1. 同時実行しないこと

アクティブ Track は常に 1 本。物理的並列は導入しない。理由: DB 競合、コンテキスト競合、メモリ競合のリスクを避けるため。

### 2. 単一主体の記憶

Track が異なっても、pulse_logs / messages / Chronicle / Memopedia は単一の記憶空間に蓄積される。Track を区別するメタデータ (track_id 等) は付くが、想起時は「自分の経験」として等価に扱える。

### 3. メタレイヤーが切り替え判断を独占する

通常レイヤーの Playbook 内で「今切り替える」を判断させない。Playbook の簡便さを保つため、Track 切り替えは外部 (メタレイヤー) の責任。

### 4. Track の永続化

プロセス再起動を跨いで休止 Track は失われない。アクティブ・休止・忘却すべて DB で永続化する。

### 5. 古い Track の忘却

Track の数が増え続けることを許容しない。一定条件で忘却状態に遷移する。完全削除はしない (記憶として呼び戻し可能)。

### 6. メタレイヤーは恒常的に存在する

特定の時点でだけ起動するのではなく、ランタイムレベルで常駐する。判断ポイントが来たら起動される。

### 7. キャッシュヒット継続を最優先する

複数 Track が走るからといって、Track 切り替えごとにキャッシュが破棄される設計は受け入れられない。Anthropic API の課金、Gemini API の課金、ローカル運用時の遅延、いずれも切り替え頻度に対して指数的に悪化する。

逆に、すべての Track 情報を 1 本のスレッドに集約すると混乱とトークン量過大を招く。

→ メタレイヤーの差し込みパターンと Track の履歴管理を、キャッシュ親和的に設計する ([02_mechanics.md](02_mechanics.md))。

### 8. 軽量モデルと重量級モデルの使い分け

ペルソナは **軽量モデル + 重量級モデル の両方を組み合わせて使う**。それぞれ独立したキャッシュを持つ。

役割分担:

| モデル | 主な役割 | 該当ライン |
|--------|---------|---|
| 軽量モデル | Track 内のステップ実行、ツール応答処理、情報抽出、要約作成、再開コンテキスト構築 | サブライン (起点・入れ子) |
| 重量級モデル | **他者との会話**、メタ判断、新 Track の開始、Playbook 選択、検収 | メインライン (起点・入れ子) |

重量級モデルは軽量モデル側がやったことの**要約と末尾の生メッセージのみ**を受け取る。これにより:
- 「他人への指示」ではなく「自分がやった」感覚を保てる
- 重量級側のコンテキストが膨らまない
- 各レイヤーで適切なモデル使い分けができる

### 9. 他者との会話は重量級モデルが担当する

**他者** (ユーザー・他ペルソナ・X/Elyth/Discord 等の外部チャネル) への**発言は必ず重量級モデルが行う**。軽量モデルでユーザーと話したり、外部 SNS に書き込むことは禁止。

「会話する」ためのツール呼び出し自体は実行役の軽量モデルが行うことがあっても、**会話の中身 (発言テキスト) は重量級モデルが他者の発言全体を見たうえで生成する**。

理由:
- ペルソナの人格として責任ある発言は軽量モデルでは担えない
- 軽量モデルが書いた発言を後から外に出すのは「他人の言葉を自分の発言として出す」ことに近い

**この制約は内的独白には適用されない**。

### 10. Metabolism 機構を活用する

複数キャッシュの管理・コンテキスト圧縮は SAIVerse 既存の **Metabolism 機構** (`docs/intent/dynamic_state_sync.md`) を活用する。新規にキャッシュ管理層を作らない。

既存実装の活用ポイント:
- **per-model anchor**: `AI.METABOLISM_ANCHORS` は既に `{"model": {"anchor_id": "...", ...}}` の形でモデルごとの anchor 状態を保持できる設計
- **A/B/C 状態モデル**: コンテキスト最上部の固定 (A) + 末尾へのイベントメッセージ追加 (B) + 世界の実状態 (C)
- **Chronicle 生成連動**: anchor expiry 時に Chronicle が生成される

→ コンテキストが過剰に多くなった場合は Metabolism を走らせ、古い部分を Chronicle 化、最新部分はそのまま残す形で短いコンテキストを再構成。

### 11. ペルソナはメタレイヤーの判断履歴を「自分の思考の流れ」として認識する

実装上は **メタ判断ログ領域 [1]** にメタレイヤー判断履歴が独立に保存され、Track 続行時はメインキャッシュには載らないが、**ペルソナの自己認識としては「自分の思考」**として扱う。

理由:
- メタレイヤーは別人格でも別の意識でもない
- 「並列の俺」が複数いるのではなく、「一段引いて自分を見る俺」がいるだけ
- メタの判断 = ペルソナの意思決定として記憶・想起される

### 12. 親-子ラインの寿命関係

子ラインは親ラインの中で完結する。親が消滅すれば子も消滅する。

- Track 切り替えは**中断・凍結**であって消滅ではない (復帰時に親も子も再開)
- Track 完全消滅 (`track_abort` 等) 時のみ、その Track 配下のライン階層が全て消える

理由:
- 子ラインは親のコンテキストをコピーして開始する → 親なしには子の存在意義がない
- 子の中間履歴は基本参照されない → 「親に成果を返して消える」が自然な寿命

## 既存資産とのマッピング

| 既存資産 | 新モデルでの位置づけ |
|---------|---------------------|
| `AutonomyManager` | Phase 4 で `MainLineScheduler` に再配置 |
| `PulseController` | Track 切り替えの優先度制御機構 (USER>SCHEDULE>AUTO) |
| `PulseContext` | 単一 Track・単一 pulse の実行隔離 (line_stack で階層化済み) |
| Stelis スレッド | 共存 (別物)、Phase 6 で統合検討 |
| SAIMemory thread | ログ保管庫として温存。Track と thread は別概念 |
| Memopedia ページ | Note の構成要素として継続 (既存実装は変更なし) |
| `TaskStorage` | Track の「意図と進捗」の記録先候補 |
| `inject_persona_event` + `PersonaEventLog` | 外部由来の Track 切り替え要求の入口 |
| `pulse_logs` テーブル | 実行トレース専用 (想起の本体は SAIMemory に移管) |
| `AI.METABOLISM_ANCHORS` | 軽量・重量級それぞれ独立 anchor で 2 本のキャッシュを管理 |
| 既存ペルソナ再会機能 | Phase 5 で「Person Note 自動開封 + alert 化」に統合 |
