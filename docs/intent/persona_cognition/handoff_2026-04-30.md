### 3. メタ判断を独白 + /spell 方式に回帰 (未コミット)

Phase C-2 のテスト中に Intent A の本質設計から逸脱していたことが判明:
- 旧実装は構造化出力 (response_schema) で `{action, ...}` JSON を返す形式
- これは intent A v0.9 不変条件 11 (= メタ判断はペルソナの自分の思考の流れ) と矛盾。Track 内のメインキャッシュに JSON が混入すると以後の発話も JSON 化する副作用がある
- マルチプロバイダ互換性も悪い (Gemini SDK は any_of のみ、OpenAI strict は anyOf 非対応、Anthropic は anyOf 制限付き)

→ **独白 + /spell 方式** (旧 legacy path と同じ) に回帰:
- `meta_judgment.json` の judge ノードを response_schema なしの自然言語応答ノードに
- ペルソナが独白の中に `/spell track_pause ...` `/spell track_activate ...` を埋め込む形式
- `meta_judgment_dispatch.py` 廃止 (Track 切替判定はスペル発動と etc. 状態遷移 hook で完結)
- `revisions.md` に v0.15 として経緯記録

### 4. TrackManager 状態遷移 Observer + scope 昇格 hook (未コミット)

メタ判断 Playbook が独白方式に戻ったので、scope `'discardable'→'committed'` の昇格機構をリファクタ:
- 旧: `_track_common._maybe_promote_meta_judgment` + `PulseContext.meta_judgment_message_id` (タイミング問題で動かず)
- 新: TrackManager に `add_status_change_observer` を追加。`_set_status` / `activate` / `pause` / `wait` / `complete` / `abort` / `resume_from_wait` の各メソッドで状態遷移後に `_notify_status_change(persona_id, pulse_id)` を発火
- SAIVerseManager 起動時に `_promote_meta_judgment_in_pulse` を hook 登録。pulse_id ベースで当該 Pulse 内のメタ判断ターンを SQL UPDATE で 'committed' 化

これで「保存ロジックは scope='discardable' で素直に書くだけ」「昇格は Track 切替起点で hook 経由」と責務分離される。

### 5. alert からの pause 禁止 (未コミット)

debug 中に発見: `TrackManager.pause` の `allowed_from` が `{STATUS_RUNNING, STATUS_ALERT}` で、alert を pending に降ろせる仕様だった。これは intent A v0.9 03_data_model.md の `running → pending` 仕様と乖離。

`alert` は「即応すべき状況」であって、対応せずに消すと「目覚ましのアラーム止めて寝続ける」現象が起きる (まはー比喩)。`allowed_from` を `{STATUS_RUNNING}` のみに修正。

`01_concepts.md` の Alert 状態セクションに「alert からの解消ルール」を新規追加 (activate / wait / abort のみ許容、pause 禁止)。

### 6. その他

- `judge prompt` の文体強化: 「ユーザーには届かない、相手宛ての会話表現禁止、独白として自分自身に語る形式」を明示 → エアが対話モードに入る現象を抑制
- `llm_io.log` に spell loop retry も記録 (元は `_dump_llm_io` がメインの LLM 呼び出しでしか動いていなかった)

### Part 1: メタ判断 Pulse の直列化キュー (タスク #25) — ✅ 完了 (2026-04-30 後半セッション)

**問題**: メタ判断 Pulse が並列実行される構造的バグ。
- AutonomyManager の background thread が定期 tick で `on_periodic_tick` を呼ぶ
- alert observer (chat thread 経由) が `on_track_alert` を呼ぶ
- 両方とも別 thread で同じ persona に対するメタ判断 Playbook を起動 → 並列実行
- 各 Pulse は自分の `fetch_tracks` snapshot を見て判断 → 「pending と思って pause したら裏で alert になっていた」のような事故

**実装** (`saiverse/meta_layer.py`):
- `__init__` で `_persona_locks: Dict[str, threading.Lock]` + `_locks_guard` を保持
- `_get_lock(persona_id)` で遅延作成
- `on_track_alert` / `on_periodic_tick` の入口で `with lock:` で取得待ち。0.5 秒以上待った場合は INFO ログ
- テスト: `tests/test_meta_layer.py` に 2 件追加 (同一ペルソナの直列化 / 別ペルソナの並列実行)

intent docs: `02_mechanics.md` §"メタ判断 Pulse は同時 1 本 (per-persona 直列化)" + `revisions.md` v0.16。

### Part 2.5 — pulse_id カラム化 (2026-05-01) — ✅ 完了

Part 2 動作確認中に `_promote_meta_judgment_in_pulse` が `OperationalError: no such column: pulse_id` で落ちることを発見。SAIMemory の messages テーブルは pulse_id 専用カラムを持たず、`metadata.tags` の `"pulse:{uuid}"` 文字列で保存していたため、SQL 設計が実装と整合していなかった。

応急処置で `json_each(metadata, '$.tags')` 経由のタグ照会に書き換え (cfb68b4)、その後 Phase 2.5 として **pulse_id 専用カラム + INDEX を追加** (1728dbf):

- `messages.pulse_id TEXT` カラム + `idx_messages_pulse_id` INDEX
- `add_message` / `_append_message` / `_store_memory` の保存経路に pulse_id を配線
- `_promote_meta_judgment_in_pulse` の SQL を `pulse_id = ?` の INDEX 利用形に戻す
- `_backfill_messages_pulse_id` で既存行をタグから自動移行 (起動時 1 回、べき等)
- 互換維持のため `metadata.tags` の `"pulse:{uuid}"` 書き込みは併存 (読み出し全経路がカラム参照に移行したら廃止)

実機 12 ペルソナで合計 4,098 件のバックフィルが正常完走。

### Part 2 実機検証で発覚した順序バグ (2026-05-01) — ✅ 修正済み (b2c0c81)

`runtime_graph.py` の Pulse 完了処理で `committed_to_main_cache` フラグが常に `False` で記録されていた。

原因: `_apply_deferred_track_ops` (`runtime_runner.py:81`) が処理完了時に `deferred_track_ops` を `clear()` するため、その**後**で `any(op.op_type == 'activate')` を評価していた私の実装は常に空 = False になっていた。

## 動作確認のポイント (Part 1 実装後)

サーバ再起動後、以下を観察:
1. **alert 経由のメタ判断** (ユーザー発話 → set_alert → メタ判断起動) と **定期 tick** が同時起動しないこと:
   - `[meta-layer] Judgment starting: persona=...` のログが時系列で**重ならない**
   - 走行中に新規トリガが来たら、現 Pulse 完了後に次が起動 (= sequential)
2. `[track_manager]` の状態遷移ログが時系列で重ならないこと
3. `[meta-judgment-promote]` ログが Track 切替起点で出る (Part 1 修正で変わるものではないが、既存機構の動作確認)
4. UX: alert 経由のユーザー応答時間が現状 1 分半 → 最悪 3 分まで伸びうる (定期 tick 待ちのため、許容範囲)

## 触らないでほしいこと (誤解しがちな点)

- `<system>...</system>` タグ + role='user' は**設計上意図的**。Gemini が messages 中途の system role を受け付けないため、全プロバイダ互換のためにこの形式を統一している。`role='system'` に直すと Gemini 互換が壊れる (`sea/runtime_llm.py` の自動ラップ箇所のコメント参照)
- メタ判断は構造化出力を使わない。`response_schema` を judge ノードに復活させると、JSON が Track メインキャッシュに混入してそれ以降の発話も JSON 化する副作用が出る (intent A 不変条件 11 違反)
- `pause` の `allowed_from` に `STATUS_ALERT` を再追加しない。alert を pause で消せると「目覚ましアラーム止めて寝続ける」現象が起きる
- メタ判断 Pulse は同時 1 本。並列禁止の構造化が必要 (Part 1 で実装予定)
