## 次セッションの目標

**Phase 3 残件: 既存 Playbook の `context_profile` / `model_type` を `line: "main"|"sub"` 仕様に翻訳する。**

現在 49 Playbook が `context_profile` を、17 Playbook が `model_type` を使っている (旧仕様)。これらは v0.9 で DEPRECATED 化されているが、コード上は許容されている。Phase 3 完了 (= 旧仕様の完全削除) の前に、まず**翻訳作業**を完遂する必要がある。

完了の判定基準:
- すべての Playbook で `context_profile` / `model_type` が消えている
- 代わりに `sub_play` ノードに `line: "main"|"sub"` が適切に乗っている
- LLM ノードは `line` フィールドを使わない (親ラインに従うのがデフォルト)
- 既存の動作 (重量級 / 軽量モデルの使い分け) が保たれていること

## 翻訳ルール (案 — 次セッションで確定する)

### `sub_play` ノード (子 Playbook 起動)

| 旧仕様 | 翻訳後 |
|---|---|
| `context_profile: "conversation"` | `line: "main"` (親メインキャッシュ継続) |
| `context_profile: "isolated"` | `line: "sub"` (別サブキャッシュへ分岐) |
| `model_type: "default"` (重量級) | `line: "main"` |
| `model_type: "lightweight"` (軽量) | `line: "sub"` |

両方が同時に書かれてる場合は `line: "sub"` を優先 (より制約が強い方)。実際の Playbook を 1 件ずつレビューしてどっちの意図だったかを確認する。

### `llm` ノード

LLM ノードは `line` フィールドを持たない (親ラインに従うのがデフォルト)。なので **削除** が基本。ただし:
- 過去に「重量級でしか動かない」前提だった LLM ノードが、親が軽量サブに変わると壊れる可能性 → 親 Playbook の `sub_play` 設定を確認しながら判断
- meta_judgment / meta_auto_full / meta_autonomy_decision のような **メタ判断系** は重量級必須なので、親 sub_play で `line: "main"` 強制が必要

### Track 種別 Playbook (track_*)

- これらは Pulse-root として走るため、自身の中の sub_play ノードのみ翻訳対象
- 親が main か sub かは AutonomyManager / MainLineScheduler が決める領域

## 必要な前提作業

### 1. 翻訳ルールの intent 確定

`02_mechanics.md` § "Playbook 起動とラインの関係" で line 仕様は確定済みだが、**翻訳ルール表** (上の表) を `phases/phase_3_lines_playbooks.md` に明記する必要がある。次セッション最初に書く。

### 2. `report_to_parent` バリデーション厳密化

現状: `can_run_as_child=true` の Playbook で `output_schema` に `report_to_parent` が無い場合、警告ログのみで通している (`README.md` 進捗表で🟡)。翻訳後に新たに sub line で動く Playbook が増えるため、**厳密化** (= ロード時に raise) するべき。最低限「警告から ERROR」に上げて検出を強める。

`sea/playbook_models.py` あたり。

### 3. `migrate_playbooks_to_lines.py` スクリプト作成

64 Playbook を手動で書き換えるのは現実的でない。半自動翻訳スクリプト:

- `context_profile` / `model_type` を読んで `line: "main"|"sub"` を導出
- `sub_play` ノードのみに `line` を書き、`llm` ノードからは context_profile / model_type を削除
- `--dry-run` で差分プレビュー
- `--apply` で書き戻し
- 各 Playbook を手動レビューする補助情報を出力 (特に「両方書かれてた」「LLM ノードに付いてた」等の判断が要る箇所をマーク)

**置き場所**: `scripts/migrate_playbooks_to_lines.py`

### 4. 各 Playbook の手動レビュー

特に複雑な親子関係を持つ Playbook (deep_research / research_task / source_* / sub_*) は機械翻訳だけだと誤訳リスクがある。1 件ずつ「親 Playbook がどう呼ぶか」「子 Playbook が何を期待するか」を確認:

```bash
# ある Playbook が誰から呼ばれるかを grep
grep -r '"playbook": "deep_research"' builtin_data/playbooks/
```

### 5. DB へのインポート

書き換え後に `python scripts/import_playbook.py --file <path>` を全 Playbook に対して実行 (または`scripts/import_all_playbooks.py`)。インポート後は DB の `playbooks` テーブルから `context_profile` / `model_type` フィールドが落ちていることを確認。

### 6. (Phase 3 完了後) 旧仕様コードの完全削除

スコープ外だが、翻訳完遂後の最終ステップ:
- `sea/playbook_models.py` から `context_profile` / `model_type` / `exclude_pulse_id` フィールドを削除
- `sea/runtime.py` などから旧仕様の参照箇所を削除
- 関連する DEPRECATED コメントを除去
- これは別 PR 推奨 (翻訳と削除の責務を分ける)

## 推奨される進め方

```
1. 翻訳ルール intent 確定 (phases/phase_3_lines_playbooks.md に追記)
   ↓
2. report_to_parent バリデーション厳密化 (sea/playbook_models.py)
   ↓
3. migrate_playbooks_to_lines.py 作成 (--dry-run で全 Playbook の差分プレビュー)
   ↓
4. 単純な Playbook 群から段階的に翻訳 → import → 動作確認
   - 第1群: track_* (5 件、構造が単純)
   - 第2群: source_* (6 件、ほぼ同形)
   - 第3群: sub_* (8 件、router 系)
   - 第4群: 複雑なもの (deep_research, research_task, agentic_chat 等)
   ↓
5. 全件翻訳完了 → 統合テスト (実機でエアと話して動作確認)
   ↓
6. (別 PR) 旧仕様コード完全削除
```

第1〜2群は機械翻訳でほぼ片付くはず。第3〜4群は手動レビュー必須。

## 触らないでほしいこと (再掲)

- `<system>...</system>` タグ + role='user' は意図的設計 (Gemini 互換)
- メタ判断は構造化出力 (response_schema) を使わない (キャッシュ汚染防止)
- `pause` の `allowed_from` に `STATUS_ALERT` を再追加しない
- メタ判断 Pulse は同時 1 本 (per-persona 直列化、Phase 2.6 で確立)
- `set_alert` の no-op (running) パスでも observer 通知は走らせる (Phase 2.6 で確立)
