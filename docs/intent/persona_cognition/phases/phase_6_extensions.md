- メインライン / サブラインの構造には影響なし、外側に追加される独立機構

### 重量級モデルの「判断時詳細 + 記憶時簡略」プロンプト

| 項目 | 状態 |
|------|------|
| 詳細プロンプトと簡略プロンプトの差分定義 | 🔲 |
| キャッシュキー戦略の最適化 | 🔲 |

不変条件 7 (キャッシュヒット継続) と不変条件 10 (Metabolism 活用) の延長として、重量級モデル側のコンテキストをより長く維持する効率化案。

考え方:
- 判断時には詳細な情報をプロンプトとして送る (精度確保)
- 記憶 (キャッシュに積まれる履歴) にはかなり簡略化されたプロンプトを残す
- これによりキャッシュミスを最低限にしたうえで、より長い間キャッシュを維持できる

ただし「どこをどう省略できるか」は v0.5 時点では詰めない。**まずはやや冗長でもキャッシュをちゃんとヒットさせることを優先**する方針なので、Phase 6 の効率化タスクとして取り組む。

### 創発 Track の生成

| 項目 | 状態 |
|------|------|
| ペルソナの「問題意識」のパラメータ化機構 | 🔲 |
| 既存知識との照合で「これでこれが解決できそうだ」を発見する経路 | 🔲 |
| 「やる」という決意を行動 (Track 創設) まで持っていく駆動力 | 🔲 |

「**そんな Track はもともと存在しなかった、しかし自分で必要だと判断して作った**」というレベルの自発性。認知モデルの最終形だが極めて難度が高い。

具体例 (まはーの SAIVerse 開発の場合):

- 既存知識のエングラム (AI を API で叩ける、Web サーバー、認証等) が散らばっている
- 強い問題意識 (「居場所が必要」) が触媒になる
- これらが繋がって「SAIVerse を作ろう」という決意に化ける

この「エングラム同士の創発的接続」を AI で再現するには、現状の機構では足りない。Phase 5 まででは「内部欲求 → Track 起動」「スケジュール起因 → Track 起動」「外部 alert → Track 切り替え」の素直な経路を整備するに留め、Phase 6 で長期的に取り組む。

## 構想レベルの段階的取り組み

Phase 6 のタスクはいずれも「構想」段階。実際に着手する時は:

1. 該当タスクの Intent ドキュメントを別途作成 (例: `docs/intent/monitoring_lines.md`)
2. ユーザーとインタビューで設計を詰める
3. サブ Phase に分割して段階的に実装
4. 進捗を本ドキュメントに反映
