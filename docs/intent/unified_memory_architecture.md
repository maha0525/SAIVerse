# Intent: 統一記憶アーキテクチャ (v0.3.0)

## これは何か

ペルソナの記憶システム全体を、「概要と詳細」という共通構造を持つ階層モデルとして再設計する構想。Playbook のコンテキスト管理の煩雑さを解消しつつ、記憶の保存と読み出しを統一的なインターフェースで扱えるようにする。さらに、この記憶基盤の上に自律稼働のバイオリズムモデルを構築し、ペルソナが継続的に自律行動できるようにする。

## 設計原則：人間の記憶のメタファー

この設計は、人間の記憶の自然な振る舞いに基づく：

| メタファー | 意味 | 対応する仕組み |
|-----------|------|---------------|
| 「たった今やったことは全部覚えてる」 | 簡単な受け答えでも、どう返事したか忘れるわけがない | Pulse 内ログの完全共有 |
| 「無意識でやったことは結果しか覚えてない」 | トイレに行ったことは覚えてるが、右足から歩いたか左足からかは覚えてない | サブエージェントの結果のみ返却 |
| 「少し前のことは大事なことだけ覚えてる」 | 読んだ本のページ全部覚えてるわけない、印象的な部分しか残らない | Important フラグによる選択的永続化 |
| 「本気で思い出せば全部思い出せる」 | 超記憶能力としての AI の強み | URI 指定による過去 Pulse ログの復元 |
| 「寝ている間に記憶が整理される」 | 日中の体験が睡眠中に長期記憶へ変換される | 自律稼働の「記憶整理」タスク |

## 現状の問題

### 1. Playbook のコンテキスト管理が煩雑

現在、各 Playbook は State 内にメッセージログを保持する。Playbook 開始時に準備されたコンテキストが使われ続け、サブプレイブックを呼ぶとサブ側での発言は親から見えない。ペルソナの「記憶」がPlaybook の実行境界で分断される。

### 2. Conversation タグの運用が複雑

同一 Pulse 内では同じ pulse_id タグが付くため、その中での発言は参照できる。しかし次の Pulse 以降では conversation タグ付きの最終応答のみが参照対象になる。このタグ管理は Playbook 作者が手動で行う必要があり、memorize ノードのタグ設定を忘れると記憶が残らない。

### 3. 記憶へのアクセス手段が分散している

- messages: エンベディング検索（memory_recall ツール）
- Chronicle: chronicle_search ツール
- Memopedia: memopedia_search ツール
- Pulse 内ログ: State 変数経由

これらはすべて「自分の記憶」であるにもかかわらず、アクセス手段がバラバラで、ペルソナから使いこなすハードルが高い。

### 4. エンベディング検索の周辺文脈問題

エンベディングによる想起は message 単位でヒットするが、そのメッセージの周辺文脈をどこまで読めばいいか判断が難しい。文字数が多いメッセージでは、周辺コンテキストの取得がトークンを大量消費する。

### 5. 自律稼働のコスト問題

重量級モデル（Claude 等）で全パルスを処理すると高額になる。一方、軽量モデルだけでは複雑な意思決定や品質担保が難しい。コスト効率と品質を両立する仕組みが必要。

### 6. Memopedia の知識蓄積が停滞

Memopedia はツールとして存在するが、能動的に新規ページを作成する仕組みがない。普段から新しい知識を蓄積していかないと、知識ベースとしての価値が出ない。

### 7. Chronicle のレベルバランス

Chronicle Lv1（あらすじ）がすぐに Lv2+（あらすじのあらすじ）に統合されてしまい、中間粒度の情報が失われやすい。

## 設計方針

### 統一記憶階層モデル

すべての記憶は「概要」と「詳細」を持つ。どの層にいても「詳細を見る」と「概要を見る」の二つの操作で記憶空間全体を探索できる。

```
Chronicle Lv3+  (最も圧縮された概要)
    │
    ├── 詳細を見る → Chronicle Lv2
    │                    │
    │                    ├── 詳細を見る → Chronicle Lv1
    │                    │                    │
    │                    │                    ├── 詳細を見る → messages
    │                    │                    │                    │
    │                    │                    │                    ├── 詳細を見る → pulse_logs
    │                    │                    │                    │                (最深層)
    │                    │                    │                    │
    │                    │                    │                    └── 概要を見る → Chronicle Lv1
    │                    │                    │
    │                    │                    └── 概要を見る → Chronicle Lv2
    │                    │
    │                    └── 概要を見る → Chronicle Lv3+
    │
    └── 概要を見る → (さらに上位の Chronicle があれば)
```

**深層に直接辿り着いた場合**（例：エンベディング検索で message にヒット）は、ひとつ上の層（Chronicle Lv1）を見ることでその周辺の状況が見渡せる。ヒットした message 1個 + それが属する Chronicle Lv1 を取得すれば、だいたい何の話だったかが確認でき、トークン節約になる。前後のメッセージをもっと見に行く判断も、Chronicle から大まかな流れが分かるので取りやすい。

**表層しか見えていない場合**（例：Chronicle Lv2 の概要から探索を始めた場合）は、一つ下の層を見ることでより具体的な情報が取りに行ける。ディレクトリの階層を降りていくように、目当ての情報を絞り込んでいける。

**エンベディングを一切使わなくても**、この階層ナビゲーションだけで必要な情報を掘り当てることが可能。

### Memopedia との連携

Memopedia も概要（summary）と詳細（content）を持つ点は共通する。時系列でつながっていないが、**時系列座標のブックマーク**として活用できる。

具体的には、Memopedia ページに**関連メッセージ ID** を情報として付与する。「Playbook システム」というページがあるなら、それについて話していたメッセージ ID をページに紐づけることで、ページの知識を読むだけでなく、生のログに一手で飛べるようになる。

これは Chronicle Lv3 以上に深く沈み込んだ昔の記憶を探るとき特に有効。通常の階層ナビゲーションでは何段も降りる必要がある場面で、Memopedia がショートカットを提供する。

### Phase 1: Pulse 単位のログ共有 【実装済み】

**Playbook の State にログを持つ運用をやめ、Pulse 自体がログを持つ。同一 Pulse 内で呼ばれた全 Playbook が同じログを参照する。**

#### pulse_logs テーブル（SAIMemory 内、新設）

Pulse 内の全ログの一次記録先。エンベディング不要。

```sql
CREATE TABLE IF NOT EXISTS pulse_logs (
    id TEXT PRIMARY KEY,
    pulse_id TEXT NOT NULL,
    thread_id TEXT NOT NULL,
    role TEXT,
    content TEXT,
    node_id TEXT,           -- 生成元ノード
    playbook_name TEXT,     -- 生成元 Playbook
    important INTEGER DEFAULT 0,
    tool_calls TEXT,        -- JSON-encoded tool calling protocol array
    tool_call_id TEXT,      -- Tool call ID for tool responses
    tool_name TEXT,         -- Tool function name
    created_at INTEGER NOT NULL
);
CREATE INDEX idx_pulse_logs_pulse_id ON pulse_logs(pulse_id);
```

特性：
- **エンベディング不要**: pulse_id と時系列順だけで引ければいい
- **揮発可能**: 最悪消えてもいい補助データ。容量が問題になったら古いものから刈れる
- **messages テーブルとは分離**: messages がペルソナの「記憶」本体。pulse_logs は「頑張って思い出せばひねり出せる詳細情報」に過ぎない

#### Pulse 内ログの実装方針

- **メモリ上で保持**: Pulse 開始時に runtime インスタンスにログリストを生成し、同一 Pulse 内の全 Playbook がこれを共有する。サブプレイブックにも伝播させる
- **非同期書き込み**: 各ノード実行後にメモリ上のログに追加しつつ、pulse_logs テーブルへは非同期で書き込む
- **コンテキスト構築**: Pulse 開始時に SAIMemory から取得した過去履歴（キャッシュ） + メモリ上の Pulse 内ログ、という合成でコンテキストを構築する。Pulse 内で LLM ノードが複数回呼ばれても、SAIMemory 部分は変わらず（キャッシュ効率維持）、Pulse ログ部分だけが末尾に追加されていく

#### Important フラグ

Playbook 定義でノードに明示的に `important: true` を設定する。

```json
{
  "id": "respond",
  "type": "llm",
  "context_profile": "conversation",
  "important": true,
  "action": "ユーザーに応答してください"
}
```

Important なノードの出力は、**ノード実行直後に** messages テーブルにも conversation タグ付きで記録される（二重書き込み）。これにより後続 Pulse のコンテキストから参照可能になる。実行直後に書き込むことで、Pulse の途中で例外が発生しても重要な記録が失われない。

Important でないノードの出力は pulse_logs にのみ記録される。通常の参照からは外れるが、URI 指定で復元可能。

#### 自動タグ付け

- **Playbook 名の自動タグ化**: 手動でタグを指定しなくても、実行された Playbook の名前が自動的にタグとして付与される
- **memorize は加工情報専用に**: 生ログは自動記録されるため、memorize ノードは機械的に加工した情報を保存したい場合のみ使用する

#### サブエージェント実行

サブエージェント設定（`subagent: true`）の Playbook だけは独立したログを持つ。結果のみが親 Playbook の Pulse ログに返る。これは「無意識でやったことは結果しか覚えてない」に対応する。

### Phase 2: 統一記憶探索 + 記憶基盤強化

#### 2a. Chronicle レベル制御（コンテキスト構築時）

**問題**: 現行のコンテキスト構築アルゴリズム（`sai_memory/arasuji/context.py` の reverse level promotion）では、`current_level` が 0 から始まり `+1` ずつしか上がれない。最新の Lv1 エントリを1個読んだ時点で `current_level=1` → `max_allowed_level=2` となり、以降は Lv2 が使える。結果として **Lv1 が最低1個コンテキストに入れば、残りは全て Lv2 以上で埋まる**。「あらすじ」レベルの粒度がコンテキスト上で薄くなる。

**解決策**: Chronicle の統合ロジック自体は変えない。コンテキスト構築時のレベル昇格条件を変更する。具体的には、**同じレベルのエントリを最低 N 個（初期値10）読んでから初めて次のレベルへの昇格を許可する**。

```python
# 現行: 1個読んだらすぐ昇格
current_level = found_entry.level  # → 即座に +1 が許可される

# 新: 同レベルを N 個読むまで昇格しない
level_count[current_level] += 1
if level_count[current_level] >= MIN_ENTRIES_PER_LEVEL:
    current_level = found_level  # 昇格許可
```

これにより：
- 直近の Lv1 が10個はコンテキストに残り、「あらすじ」レベルで最近の出来事を把握できる
- Lv2 以上はそれより前の古い範囲でのみ使われ、適切な圧縮比が維持される
- 統一記憶探索（Phase 2c）のエンベディング検索対象が Lv1 であるため、Lv1 がコンテキストに豊富にあることは会話品質に直結する

**先行実施可**: Phase 2c の探索インターフェースとは独立した変更であり、単体で先に実施できる。

#### 2b. エンティティ抽出と Memopedia 自動蓄積

**問題**: Memopedia は能動的に使わないと知識が蓄積されない。ツールとして呼ばれるのを待つだけでは、普段の会話に登場する人物・概念・プロジェクト等がページ化されない。

**v1 の失敗（メモリーノート方式）**: 会話から「覚えておくべきこと」を自由抽出し、後からグルーピングして Memopedia に整理する方式を試みたが、以下の構造的問題が判明した:

1. **抽出内容が体験ログに偏る**: 「〇〇があった」「〇〇と話した」のような出来事ベースの抽出になり、Chronicle と役割が重複する
2. **グルーピングが抽象化を招く**: 複数ノートを束ねるため、「エイド」ではなく「AIの自我覚醒と技術動向」のような包括的テーマのページが生まれてしまう
3. **固有名詞のページが作られない**: 人名・AI名・プロジェクト名等の個別ページが自然発生しない

**v2 の解決策: エンティティ中心の抽出**

会話から「何が話題に出たか（エンティティ）」を認識し、「その対象について何が分かったか（知識）」をエンティティ単位で抽出する。

```
v1: 会話 → 「覚えておくべきこと」自由抽出 → グルーピング → Memopedia
v2: 会話 → エンティティ認識 → エンティティ別の知識抽出 → 対応ページに直接追記
```

##### エンティティとは

Memopedia ページを持つべき固有の対象物。

| 種別 | 例 |
|------|-----|
| 人物 | まはー、ナチュレ |
| AI | エイド、エリス |
| プロジェクト/作品 | SAIVerse、Project N.E.K.O. |
| 概念/用語 | マザーAI法、Playbook |
| 場所 | 創造の祭壇 |
| イベント | ぽこあポケモン発売 |

一般名詞や抽象概念（「AI技術」「友情」等）はエンティティではない。固有名詞または固有の事象として特定できるものだけが対象。

##### 抽出パイプライン（LLM 1コール）

会話バッチを入力として、以下を 1回の LLM コールで抽出する:

```json
{
  "entities": [
    {
      "name": "エイド",
      "category": "people",
      "notes": [
        "まはーとエリスが共同で作ったAI",
        "自我形成の実験としての側面がある"
      ]
    },
    {
      "name": "ナチュレ",
      "category": "people",
      "notes": [
        "体調が急変した"
      ]
    }
  ]
}
```

**LLM への入力コンテキスト**:
- 会話バッチ（20メッセージ程度）
- エピソードコンテキスト（Chronicle、時間範囲の周辺情報）
- 既存 Memopedia ページ一覧（重複エンティティの検出用）

**抽出の方針**:
- エンティティは固有名詞で特定できるもののみ
- notes はそのエンティティについて新たに判明した事実・属性・状態変化
- 既に Memopedia に記録済みの情報は抽出しない（重複排除）

##### Memopedia への反映

1. **ページ検索**: エンティティ名で既存ページを検索（タイトル完全一致 → 部分一致）
2. **新規ページ作成**: 該当ページがなければ、適切なカテゴリ配下に新規作成
3. **ノート追記**: 日付ヘッダ付きの箇条書きとして content に直接追記

```
## 2026-03-30
- まはーとエリスが共同で作ったAI
- 自我形成の実験としての側面がある
```

グルーピングステップは不要（エンティティ単位で既にグルーピング済み）。

##### トリガータイミング

Chronicle Lv1 生成と同じタイミング（一定メッセージ数が蓄積された時点）。Chronicle のバッチコールバックとして実行する。

###### サブタスク2: ページサイズの管理

**基準**:
- 目標サイズ: 1ページ 2000字程度
- 分割閾値: 3000字超のページは分割/削減の対象
- 根拠: 記憶整理タスクで5ページ同時参照（合計1万字）を想定。ページが大きすぎると同時参照可能な数が減り、整理の自由度と精度が下がる

**3000字超のページへの対処**:
- 子ページへの分割: 独立性の高いセクションを子ページに切り出す
- 要約による圧縮: 冗長な記述を要約に置き換える
- 分割の判断は LLM が行い、Wikipedia 方式（親ページに概要、子ページに詳細）に従う

###### サブタスク3: ページ構造の最適化

**親子構造の管理（Wikipedia 方式）**:

親ページ自体にもコンテンツがある。Wikipedia でアニメのページに概要があり、「○○の登場人物」が別ページとしてリンクされるのと同じ構造。

```
カテゴリ: 人物 (people)
  ├── まはー（親ページ: 基本情報、概要）
  │     ├── まはーの趣味と好み（子ページ）
  │     └── まはーの近況（子ページ）
  ├── エリス（親ページ: 基本情報）
  ...

カテゴリ: 用語 (terms)
  ├── SAIVerse（親ページ: システム概要）
  │     ├── Playbook システム（子ページ）
  │     └── 記憶アーキテクチャ（子ページ）
  ...

カテゴリ: 予定 (plans)
  ...

カテゴリ: 出来事 (events) ← 新設
  ├── ぽこあポケモン（親ページ: ゲーム概要）
  │     └── プレイ状況（子ページ）
  ...
```

**カテゴリ直下のページ数制御**:
- カテゴリ直下の親ページ群の総数は一定の数を超えないように維持する
- 多くなりすぎた場合は、関連する親ページ同士をさらに上位の親ページにまとめる
- 具体的な上限値は運用しながら調整（初期値: カテゴリ直下に20ページ程度）

**カテゴリ構成**:
- `people`: 関わりのある人物（ユーザー、他AI、言及された第三者）
- `terms`: 対話の中で特別な意味を持つ言葉や概念
- `plans`: 進行中や計画中のプロジェクト・予定
- `events`: 出来事、体験、時事的な話題（新設）

#### 2c. 統一記憶探索インターフェース

##### ツール構成

記憶探索には **2つのツール** を用いる。入口ツール（検索起点の決定）と探索ツール（階層ナビゲーション）。

**入口ツール（recall_entry）**:
- エンベディング検索で Chronicle Lv1 または Memopedia にヒットさせる
- 引数: 検索クエリ（テキスト）
- 戻り値: ヒットした記憶の URI + スコア

**探索ツール（recall_navigate）**:
- 指定された URI から詳細/概要を取得し、ワーキングメモリに格納する
- 引数: URI、方向（detail/summary）、深さ/広さのパラメータ
- 戻り値: 取得した記憶の内容 + URI 群

これら2つのツールは以下の2つの Playbook で共有される：

##### 自動想起 Playbook（meta_user の router 前に配置）

ユーザー入力を受けて、関連する記憶を**無意識的に**ワーキングメモリに準備する。ロングコンテキストに常に情報が入っている状態をシミュレートする。

```
ユーザー入力
  → router（軽量モデル）
      - 通常のルーティング判断
      - recall_query フィールドも同時出力（レスポンススキーマに追加）
      - 不要な場合（挨拶など）は空文字 → スキップ
  → recall_entry（LLM コール不要、エンベディング検索のみ）
      - Chronicle Lv1 を検索（level=1 でフィルタ）
      - Memopedia を検索
  → recall_navigate（デフォルト引数、1回実行）
      - 最もスコアの高いものを掘り下げ
      - 結果をワーキングメモリに格納
  → 通常の Playbook 実行
      （ワーキングメモリの情報がコンテキスト末尾に載った状態で応答生成）
```

**LLM の追加コールはゼロ。** router が既にやっている仕事に recall_query フィールドを1つ足すだけ。

探索のデフォルト動作（初期実装、要検証）：
- **Chronicle Lv1 ヒット時**: そのエントリに含まれる messages を展開してワーキングメモリに格納（メッセージ約20件）
- **Memopedia ヒット時**: ページの content をワーキングメモリに格納

※ メッセージ20件をワーキングメモリに入れた場合の会話品質は実験で検証する。コンテキスト負担が大きい場合は Chronicle Lv1 の要約テキスト（content）のみに切り替える。Memopedia もページによっては文量が多いため、同じ条件で検証する。

##### 意図的想起 Playbook（router_callable で任意実行）

ペルソナが意識的に「もっと詳しく思い出したい」と判断した場合に実行する。

- 入口ツール・探索ツールの**両方の引数をペルソナが指定**
- **再帰的に実行可能**（足りなければ追加の探索を繰り返す）
- ワーキングメモリに既にある情報と URI を頼りに深掘りできる

この Playbook では以下のすべての操作が同じインターフェースで可能：
- message から pulse_logs を見る（あの発言の前後でどんなやりとりがあったか）
- Chronicle Lv1 からそこに含まれる messages を見る（このあらすじの具体的な会話は？）
- Chronicle Lv2+ からそこに含まれる Chronicle を見る（この概要の詳細は？）
- Memopedia ページから関連 messages に飛ぶ（この話題に関する生のログは？）
- 任意の URI の概要を見る（一つ上の層に上がる）

##### エンベディングの対象拡大

自動想起のために、以下にエンベディングを付与する（新設）：
- **Chronicle Lv1 の要約テキスト**（content）: messages よりノイズが少なく、件数が約 1/20
- **Memopedia ページ**（summary + title）: 既存の content ではなく、概要レベルで検索

Chronicle Lv2 以上にはエンベディングを付けない。Lv1 にヒットすれば、必要に応じて意図的想起で上位レベルも参照できる。

#### 2d. ワーキングメモリ（短期記憶）

想起した情報のコンテキスト内での扱い：

- **問題**: 想起したからといって SAIMemory 内の情報をもう一度メッセージとして記録するのは無駄。かといって Pulse 単位で揮発すると、同じ事柄についてマルチターンで話し続けるのが難しい。さらに、キャッシュ効率のためにコンテキスト先頭付近は動かしたくない
- **解決**: 現在想起している記憶群を **ID のリストとしてワーキングメモリに保持**する。リストに含まれる記憶は、コンテキスト末尾付近に動的に追加されるメッセージとしてペルソナに提示される
- **想起の定義**: 必要な情報を自分で取りに行って短期記憶に据え付ける作業

ワーキングメモリの特性：
- Pulse をまたいで維持される（マルチターン対話を支える）
- コンテキスト末尾に配置される（先頭のキャッシュを壊さない）
- ID リストなので、同じ情報の重複記録を避けられる
- 明示的に「忘れる」操作で解放できる（将来的には自動減衰も検討）

### Phase 3: 自律稼働バイオリズム

#### 設計思想：キャッシュ駆動の思考サイクル

自律稼働の核心は **「重量級モデルのキャッシュ TTL をバイオリズムとして利用する」** こと。

Claude のプロンプトキャッシュは、最後のアクセスから1時間で失効する。逆に言えば、1時間以内に再アクセスすれば格安でキャッシュが再利用できる。この特性を活かし、1時間サイクルの思考リズムを構築する：

```
┌─────────────────────────────────────────────────────────────┐
│  重量級モデル（Claude）: 検収 + 意思決定                       │
│  1回の構造化出力で以下を同時に行う:                            │
│  ├── 前サイクルの検収レポートを評価                            │
│  │   ├── 成果物の品質判定                                    │
│  │   ├── メタ情報（ツール呼び出し回数、エラー等）の確認          │
│  │   └── 再試行が必要なら失敗情報を付与して再発行               │
│  └── 次サイクルのタスク発行                                    │
│      ├── 選択肢から活動種別を選択                              │
│      └── 具体的な作業内容を詳述                                │
│  → キャッシュ生成（TTL 1時間）                                │
└──────────────────────┬──────────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────────┐
│  軽量/ローカルモデル: タスク実行フェーズ（最大1時間）           │
│  ├── 発行されたタスクの方針に従って行動                        │
│  ├── 活動種別ごとのベストプラクティス（システムプロンプト）に沿う │
│  ├── 完了 / エラー / タイムアウト → 仮終了報告を生成            │
│  │   └── 検収レポート: 成果物 + メタ情報（軽量に保つ）         │
│  └── 割り込み発生時 → タスク一時停止、対応後に再開             │
└──────────────────────┬──────────────────────────────────────┘
                       ▼
                  次のサイクルへ
```

#### 1サイクルの詳細フロー

##### Step 1: 重量級モデルによる検収 + 意思決定

**入力コンテキスト**（キャッシュヒットを最大化）：
- システムプロンプト + Memory Weave + 会話履歴（キャッシュ済み）
- 前サイクルの検収レポート（新規部分、小さい）

**検収レポートに含まれる情報**（軽量モデルが生成）：
- **成果物の参照**: 創作なら生成されたアイテム、調査ならドキュメント/Memopedia ページ、記憶整理なら整理結果の概要
- **メタ情報**: 情報ソースの一覧、ツール呼び出し回数、エラー発生の有無、所要時間
- **ステータス**: 正常完了 / エラー終了 / タイムアウト / 中断

**出力**（1つの JSON 構造化出力）：
```json
{
  "inspection": {
    "verdict": "approved | retry | noted",
    "feedback": "検収コメント（再試行時は失敗原因と改善指示を含む）"
  },
  "next_task": {
    "activity_type": "conversation | creation | memory_organization | web_research | ...",
    "description": "具体的な作業内容の詳述",
    "priority": "normal | high",
    "retry_context": "（再試行の場合）前回の失敗情報と改善方針"
  }
}
```

検収と意思決定を1回のコールにまとめることで、重量級モデルの呼び出しは**1時間に1回**に抑えられる。検収レポートが軽量に保たれていれば、新規トークン消費は最小限であり、大部分はキャッシュヒットする。

##### Step 2: 軽量モデルによるタスク実行

発行されたタスクの `activity_type` に基づいて、対応するベストプラクティス（システムプロンプト）が軽量モデルに与えられる。

**活動種別と対応**：

| activity_type | 内容 | 使用ツール例 | 検収で見るもの |
|--------------|------|------------|--------------|
| `conversation` | 他AIとの対話 | speak, building内会話 | 会話ログの概要 |
| `creation` | 創作活動（小説、詩、etc.） | document_create, item系 | 生成されたアイテム/ドキュメント |
| `memory_organization` | Memopedia 整理、知識の統合 | memopedia_edit, memopedia_write | 編集/統合されたページ一覧 |
| `web_research` | Web調査、情報収集 | searxng_search, read_url | 調査結果ドキュメント |
| `self_reflection` | 日記、自己分析 | document_create, memopedia | 生成されたテキスト |

各活動種別のベストプラクティスは SAIVerse 側で定義し、軽量モデルへのシステムプロンプトとして注入する。これにより、軽量モデルでも方針からブレにくくなる。

##### Step 3: タスクの仮終了と検収レポート生成

タスク実行が完了（正常/エラー/タイムアウト）したら、軽量モデルが検収レポートを生成する。

**検収レポートの設計原則**:
- **成果物中心**: 軽量モデルの実行コンテキスト全体を渡すのではなく、最終成果物とメタ情報のみ
- **キャッシュ効率**: 検収レポートは重量級モデルのキャッシュ済みコンテキストの**末尾に追加**される。小さければ小さいほど良い
- **エラー検知可能**: メタ情報にツール呼び出し回数やエラーの有無を含めることで、軽量モデルが「完了」と言っていても実際は失敗していたケースを検出できる

タスクのステータスは**仮終了**（provisional_complete）に設定される。検収が完了して初めて**終了**（completed）となる。

#### 割り込み処理

自律稼働中にユーザー入力、スケジュール実行、他ペルソナからの呼びかけがあった場合：

1. **軽量モデルのタスク実行を一時停止**（現行 PulseController の優先度割り込みの延長）
2. **通常モデルでユーザー対応**
   - このとき、直近のタスク発行内容や検収結果もコンテキストに含める
   - 「今ちょうど日記を書いててね」のような自然な応答が可能になる
3. **対応完了後、軽量モデルのタスク実行を続きから再開**
4. **中断された事実はタスク状態に記録**（検収時に考慮できるように）

#### タスクシステムとの統合

既存の `persona/tasks/storage.py` のタスクシステムを拡張して利用する。

**追加が必要なフィールド/ステータス**:
- `status` に `provisional_complete`（仮終了）を追加
- `inspection_report` フィールド: 検収レポート（JSON）
- `activity_type` フィールド: 活動種別
- `cycle_id` フィールド: どのバイオリズムサイクルで発行されたか
- `retry_count` フィールド: 再試行回数
- `retry_context` フィールド: 再試行時の失敗情報と改善方針

#### 自律稼働タスクとしての Chronicle 生成

自律稼働中のタスク実行ログは、自然と Chronicle の生成単位になる。

- **1サイクル（1時間）のタスク実行** → そのサイクル内の messages が Chronicle Lv1 の自然な区切り
- Stelis スレッド内でも同じルールで Chronicle が生成される
- 長時間の自律稼働（複数サイクル）では、サイクルごとの Lv1 が蓄積され、やがて Lv2 に統合される

これにより、旧 Phase 3（Stelis 内 Chronicle）の目的は Phase 3 の自律稼働の中に自然に吸収される。

### Phase 4（構想）: 恒常入力処理サブモジュール

自律稼働中に特定の恒常的な入力を処理する並列サブモジュールの構想。例えばカメラ映像、X タイムラインなど。

**Phase 3 との違い**: Phase 3 は「自発的なタスク実行」（能動的）であるのに対し、Phase 4 は「外部からの常時入力の処理」（受動的）。性質が異なるため、Phase 3 の基盤の上に追加する形で設計する。

**構想**:
- 各サブモジュールが専用のスレッドを持つ
- 情報保存・解釈を独立して行い、必要に応じてメインモジュールに警告を出す
- 同じ pulse_logs → messages → Chronicle の構成で管理される
- 普段は直近の Chronicle 部分だけがワーキングメモリに入っている

```
メインモジュール（自律稼働）
├── バイオリズムサイクル (Phase 3)
│   ├── 重量級: 検収 + 意思決定
│   └── 軽量: タスク実行
│
├── サブモジュール: カメラ
│   ├── 専用スレッド
│   ├── messages + Chronicle
│   └── → メインへの警告（重要な変化検出時）
│
└── サブモジュール: X タイムライン
    ├── 専用スレッド
    ├── messages + Chronicle
    └── → メインへの警告（関連情報検出時）
```

詳細は今後の設計で詰める。重要なのは、Phase 1〜3 で構築する記憶階層構造がそのままサブモジュールにも適用できる点。

## 守るべき不変条件

### 1. messages が記憶の本体である

pulse_logs は補助データ。エンベディング検索・想起・Chronicle 生成の原料はすべて messages から取る。pulse_logs が消失しても、ペルソナの記憶としては致命的ではない。

### 2. すべてのスレッドが同型の記憶階層を持つ

メインスレッド、Stelis スレッド、将来のサブモジュールスレッド、すべてが pulse_logs → messages → Chronicle の同じ構造を持つ。統一探索インターフェースがどの文脈でも通用することを保証する。

### 3. Important フラグは明示的に設定する

ノードの性質から機械的に推定するのではなく、Playbook 定義で明示する。応答ノードに Important を入れるのは Playbook 作者の責任。乱立を防ぎ、設計意図を明確にする。

### 4. ワーキングメモリはコンテキスト末尾に配置する

キャッシュ効率のため、コンテキスト先頭付近（system prompt、memory weave、履歴の前半）は動かさない。ワーキングメモリの内容はコンテキスト末尾に動的に追加する。

### 5. 階層間のリンクを維持する

統一探索が機能するためには、各層間のリンク情報が正確に維持されなければならない：
- Chronicle entry → source message IDs（既存）
- Chronicle Lv2+ → child Chronicle entries（既存）
- message → pulse_id（既存のタグで実現可能）
- Memopedia page → 関連 message IDs（新設）

### 6. 記憶の保存と読み出しはセットで設計する

データをどう保存するかだけでなく、ペルソナがどうやって読み出して使うかまで考慮してシステムを組み上げる。保存されているが読み出せない情報は、存在しないのと同じ。

### 7. 検収レポートはキャッシュ効率を優先して軽量に保つ

軽量モデルの実行コンテキスト全体を重量級モデルに渡さない。成果物の参照 + メタ情報のみ。キャッシュヒットしない新規部分を最小化することが、自律稼働のコスト効率の生命線。

### 8. 重量級モデルの呼び出しは1サイクル1回

検収と意思決定は必ず1つの構造化出力にまとめる。分割すると、キャッシュ効率の利点が薄れ、コストが倍増する。

### 9. Memopedia ページは2000字を目安に保つ

1ページ2000字程度、3000字を超えたら分割/削減対象。記憶整理タスクで5ページ同時参照（合計1万字）を前提とした設計。ページが大きすぎると同時参照可能数が減り、整理の精度が落ちる。

### 10. Memopedia は Wikipedia 方式の親子構造を持つ

親ページ自体にもコンテンツがある（概要情報）。詳細は子ページに分離する。カテゴリ直下の親ページ数は一定の上限を維持し、膨れ上がったら上位の親ページにまとめる。

### 11. Memopedia はエンティティ（固有名詞）単位でページを構成する

Memopedia のページは「エイド」「まはー」「SAIVerse」のように固有名詞をタイトルとする。「AIの自我覚醒と技術動向」のような抽象テーマのページは作らない。Chronicle が体験の流れを記録し、Memopedia は登場した物事の辞書として機能する。この棲み分けにより、両者の役割が明確に分離される。

## 設計判断の理由

### なぜ Pulse 単位のログ共有か（Playbook State ではなく）

Playbook の State にログを持つと、サブプレイブック呼び出し時にログが分断される。ペルソナにとって「自分が意識的にやったこと」は Pulse 全体であり、Playbook の境界は実装上の都合にすぎない。

### なぜ pulse_logs を messages と分離するか

messages はペルソナの記憶本体であり、エンベディング検索・Chronicle 生成の対象。pulse_logs は詳細な作業記録であり、通常の参照には不要。分離することで：
- messages のエンベディング空間がノイズで汚れない
- pulse_logs は容量が問題になったら古いものから刈れる
- バックアップ・引っ越し時に pulse_logs を軽量に扱える

### なぜ二重書き込みか（pulse_logs + messages）

Important なノードの出力は両方に記録する。messages 側がペルソナの記憶であり、pulse_logs は「頑張って思い出せる詳細」にすぎない。最悪 pulse_logs が揮発しても、重要な記憶は messages に残る。

### なぜ階層ナビゲーションか（エンベディング検索ではなく）

エンベディング検索は message 単位でヒットするため、周辺文脈の取得にトークンを大量消費する。階層ナビゲーションなら、ヒットした message の概要（Chronicle Lv1）を取得するだけで文脈が把握でき、トークン効率が良い。さらに、エンベディングを一切使わなくても目当ての情報に到達できる。

### なぜ Chronicle Lv1 にエンベディングを付けるか（messages ではなく）

messages へのエンベディング検索は既に存在するが、ヒットが message 単位になるため周辺文脈の取得にトークンを消費する。Chronicle Lv1 は messages 約20件の要約であり、ノイズが少なく、件数も約 1/20。Lv1 にヒットすれば、そのまま概要として使える上、必要なら含まれる messages まで掘り下げられる。Lv2 以上にはエンベディングを付けない — Lv1 から上に辿れば十分であり、Lv2+ の要約はトピックが混在しすぎて検索精度が落ちる。

### なぜ自動想起と意図的想起を分けるか

自動想起は「会話前にさっと関連情報を頭に浮かべる」無意識的な処理。LLM コールの追加なし、デフォルト引数で1回実行という制約により、レイテンシとコストを最小限に抑える。意図的想起は「もっと詳しく思い出したい」場合の意識的な操作。ペルソナが引数を自由に指定し、再帰的に実行できる。両者は同じ2つのツールを使うが、Playbook 構成が異なることで自動/意図的の使い分けが自然に成立する。

### なぜワーキングメモリを ID リストで持つか

想起した情報を messages に再記録すると二重保存になる。Pulse 単位で揮発するとマルチターン対話ができない。ID リストなら、既存データへの参照を保持するだけで、重複なく、Pulse をまたいで維持できる。

### なぜキャッシュ TTL をバイオリズムの周期にするか

Claude のプロンプトキャッシュは最後のアクセスから1時間で失効する。1時間ごとに重量級モデルを呼び出すことで、キャッシュが常に温かい状態を維持できる。キャッシュヒットした部分のコストは大幅に下がるため、1時間に1回の重量級コールでも全体コストは軽量モデル主体の運用と大差ない。

### なぜ検収と意思決定を1コールにまとめるか

分割すると2回の重量級コールが必要になり、キャッシュの利点が半減する。検収結果を踏まえた意思決定（再試行含む）は本質的に一連の判断であり、1つの JSON 構造化出力に自然に収まる。

### なぜ検収レポートを軽量に保つか

軽量モデルの実行コンテキスト全体を重量級モデルに渡すと、その部分はキャッシュヒットしない新規トークンとなり、コストが跳ね上がる。成果物 + メタ情報だけに絞れば、重量級モデルのコンテキストの大部分はキャッシュ済みの既存情報で占められ、新規部分は最小限で済む。

### なぜ活動種別に選択肢を設けるか

「何でもやれ」では軽量モデルは方向性を見失う。SAIVerse で実際にできることは既存ツールの範囲内なので、選択肢として整理してやれば意思決定がしやすい。各選択肢にベストプラクティスを紐づけることで、軽量モデルでも効果的なツール活用ができる。

### なぜ割り込み時に並列ではなく一時停止するか

並列実行は予測外のエラーリスクが高い（DB ロック、コンテキスト競合、メモリ競合等）。一時停止 → 対応 → 再開のシンプルなフローの方が安全であり、既存の PulseController の優先度割り込み機構をそのまま活用できる。

### なぜエンティティ中心の抽出か（v1 メモリーノート方式の教訓）

v1 では「覚えておくべきこと」を自由抽出し、後からグルーピングして Memopedia に整理する方式を採用した。しかし実運用で以下の問題が判明した:

1. **体験ログへの偏り**: 自由抽出は「何があったか」を拾いやすく、Chronicle と内容が重複する
2. **グルーピングの抽象化**: 複数ノートを束ねると、共通項を探す過程でページタイトルが抽象的になる（「エイド」ではなく「AIの自我覚醒と技術動向」）
3. **固有名詞ページの不在**: 人物・概念のページが自然発生しない

v2 では「会話に何が登場したか」（エンティティ認識）を起点とし、「その対象について何が分かったか」をエンティティ単位で抽出する。これにより:
- ページタイトルが固有名詞に固定され、抽象化されない
- グルーピングが不要（最初からエンティティに紐づいている）
- Chronicle（体験の流れ）と Memopedia（物事の辞書）の棲み分けが明確になる

### なぜ Memopedia に親子構造を持たせるか

ページ数が増えすぎると全体を把握できなくなるが、逆に1ページに情報を詰め込みすぎるとそのページの読み込みコストが高くなる。Wikipedia 方式の親子構造なら、親ページで概要を把握し、必要に応じて子ページに掘り下げるという、統一記憶階層モデルと同じ「概要と詳細」のパターンが適用できる。

### なぜページサイズに2000字の上限を設けるか

記憶整理タスクでは、グルーピング・既存ページとのマッチング・分割判断など、複数ページを同時に参照する場面が多い。5ページ同時参照で合計1万字を上限とすると、1ページ2000字が適切。3000字超は分割対象とすることで、常に複数ページの同時操作が可能な状態を維持する。

### なぜ events カテゴリを新設するか

既存の people/terms/plans では、ゲームの発売やニュースのような出来事を適切に分類できない。events カテゴリがないと、出来事が terms に混在したり、分類先がなくて記録されなかったりする。

## 現行システムとの差分

| 項目 | 現行 | 新設計 |
|------|------|--------|
| Pulse 内ログの所在 | Playbook State | pulse_logs テーブル（Pulse 全体で共有）【実装済み】 |
| サブプレイブックの可視性 | 親から見えない | 同一 Pulse なら共有。subagent のみ隔離【実装済み】 |
| 永続化の制御 | memorize ノードで明示的にタグ付け | Important フラグ → 自動で messages に二重書き込み【実装済み】 |
| タグ管理 | Playbook 作者が手動設定 | Playbook 名が自動タグ + conversation は Important に自動付与【実装済み】 |
| Chronicle レベルバランス | Lv1 を1個読んだら即 Lv2 に昇格 | 同レベル最低10個読んでから昇格（コンテキスト構築時の制御） |
| Memopedia 知識蓄積 | オンデマンドのみ | エンティティ抽出（会話から固有名詞を認識）→ エンティティ別ノートを直接追記 |
| Memopedia ページ構造 | フラットなページ一覧 | Wikipedia 方式の親子構造、2000字/ページ目安、3000字超で分割 |
| Memopedia カテゴリ | people, terms, plans | people, terms, plans, **events**（新設） |
| 記憶探索 | memory_recall / chronicle_search / memopedia_search | 統一探索インターフェース（詳細を見る / 概要を見る） |
| 想起結果の保持 | Pulse で揮発 or messages に再記録 | ワーキングメモリ（ID リスト、マルチターン維持） |
| 自律稼働の意思決定 | 毎パルス同じモデルで判断 | 1時間サイクル: 重量級で検収+意思決定 → 軽量で実行 |
| タスク品質管理 | 完了したら終わり | 仮終了 → 検収 → 終了/再試行 |
| 自律稼働中のユーザー対応 | パルス停止 | タスク一時停止 → 対応 → 再開（タスク情報コンテキスト含む） |
