# Intent: ペルソナ認知モデル — 行動の線とメタレイヤー

**ステータス**: 確定 v0.14（用語の3軸独立化 / 7層ストレージモデル / メタ判断の分岐＋破棄/来歴化フロー / メタ判断ログ独立領域）
**作成**: 2026-04-25
**改訂**: 2026-04-25 v0.2 → v0.3 → v0.4 → v0.5 → v0.6 → v0.7 → v0.8 → v0.9 → 2026-04-28 v0.10 → v0.11 → v0.12 → v0.13 → 2026-04-29 v0.14

## これは何か

ペルソナが**複数の進行中「行動の線」**を持ちながら、**単一の主体**として動く認知モデルを定義する。SAIVerse の上位概念として、自律稼働・応答待ち・並列タスク・割り込み処理・ペルソナ再会の全てが立脚する基盤。

具体例: 人間が日常生活で経験する「12時まで掃除、1時までご飯、1時半から SAIVerse 開発」のような複数の進行中タスクを行き来する状態。各時点でアクティブなのは1本だが、再開のたびに「直前まで何をやっていたか」を思い出している。複数の俺がいるわけではない。

## これは何でないか

- 物理的な並列実行（同時に複数 Playbook を動かす）ではない
- 単純な中断・再開機構ではない（v0.3.0 Phase 3 `AutonomyManager` の単一線モデルの上位拡張）
- マルチパーソナリティではない。**単一主体**が複数の文脈を持つだけ

## なぜ必要か

### 個別機能の特化を避け汎用基盤を先に作る

`feedback_generic_foundation_first.md` の原則に基づく。以下は本来同じパターンの異なる現れ方だが、現在は個別機能として分散している：

- ユーザーへの応答待ち（MCP Elicitation の動機）
- ペルソナ間の会話の返答待ち
- ペルソナ再会時の過去会話・知識復元（既に実装済み、コミット `5077028`, `a15bbb3`）
- Kitchen の長時間処理完了待ち
- スケジュール時刻まで保留される作業
- ユーザー割り込みによる自律行動の中断・復帰

これらを「行動の線」と「線間の切り替え」という1つのモデルで統一する。既存のペルソナ再会機能はこのモデルの**特殊例として既に B フロー（後述）の雛形を持つ実装**であり、汎用化の出発点となる。

### v0.3.0 中心軸との関係

`unified_memory_architecture.md` v3 (2026-04-07) の Phase 3 で導入された `AutonomyManager` は「単一の自律ループ + ユーザー割り込みでの一時停止」モデル。本ドキュメントはこの上位レイヤーとして「複数線が併存し、メタレイヤーがアクティブを選ぶ」モデルを定義する。

AutonomyManager は半ば取り壊して新メタレイヤーへ統合する方針（インタビュー Q2 結果）。Phase 3 の作業成果を破棄するわけではないが、責務はメタレイヤーへ移管する。

## 設計原則：人間の生活のメタファー

| メタファー | モデル上の意味 |
|-----------|---------------|
| 「複数のことを並行して進めている」 | 複数の行動の線が併存する |
| 「ある瞬間にやれるのは1つだけ」 | アクティブ線は常に1本 |
| 「再開したら直前まで何をしていたか思い出している」 | 線の切り替え時に文脈が復元される |
| 「全部俺の経験」 | 統一された記憶（線をまたぐ pulse_logs / messages） |
| 「ひと段落するまでは中断したくない」 | メタレイヤーが切り替えタイミングを判断 |
| 「忘れた頃にふと思い出す」 | 古い線の忘却状態と明示的な呼び戻し |

## 用語定義

### 行動の線 (Action Track / 単に「トラック」)
進行中の作業文脈の単位。SAIVerse 開発、ユーザーとの会話、特定タスクの遂行、応答待ち等、それぞれ1本のトラックとなる。トラックは明示的に閉じられるまで存在し続ける。

各トラックは固有のメッセージ履歴を持つ（実装は thread の置き換え、後述）。

### アクティブトラック (Active Track)
現在ペルソナが実行している唯一のトラック。ペルソナの計算リソース（pulse 実行、tool 呼び出し）はこのトラックに向けられる。

### 休止トラック (Dormant Track)
進行中だが現在動いていないトラック。状態（保留理由、再開条件、文脈スナップショット）を保持する。

### メタレイヤー (Meta Layer)
トラックの選択・切り替えを判断する観察視点。**通常レイヤーの上位**で、どのトラックをアクティブにするかを決める。

重要: メタレイヤーは別人格でも別の意識でもない。**同一の主体が、自分の活動を一段引いて見る視点**。

メタレイヤーは Playbook と非同期で動作する全体マネジメント役で、ランタイムレベルで常駐する（インタビュー Q3 結果）。

### トラックの切り替え (Track Switch)
アクティブトラックを別のトラックに変更する操作。メタレイヤーがトリガーする。詳細は「主要な仕組み」セクションの A/B フローを参照。

### 忘却状態 (Forgotten State)
古い休止トラックが「普段は思い出さない」状態に遷移したもの。明示的に呼び出さない限り、メタレイヤーの判断対象から外れる。

### 永続 Track と一時 Track （v0.9 で整理）

Track には「ペルソナの核として永続するもの」と「目的に応じて作って完了するもの」がある。

**永続 Track** (`is_persistent=true`):
- 完了 / 中止に遷移しない
- ペルソナのライフタイム中ずっと存在する
- ペルソナの「核」を構成する

種類:
- **対ユーザー会話 Track**: 各ユーザーごとに 1 個。そのユーザーとの長期的な関係性の文脈
- **交流 Track**: ペルソナに 1 個。他ペルソナとの会話を扱う特殊 Track。output_target=`building:current`（今いる場所）で動作。Building ごとに Track を作らないため、移動しても継続。「自室で作業中に他ペルソナが訪ねてきた」のような場面にも自然に対応

**一時 Track** (`is_persistent=false`):
- 完了 / 中止可能
- 自律行動、プロジェクト遂行、外部通信等

### Alert 状態 （v0.9 で新規導入）

Track の状態の一つ。`pending` と `running`/`waiting` の中間的な位置づけで、**可及的速やかな対応が必要**を示す。

| 状態 | 意味 | 優先度（メタレイヤー判断時） |
|------|------|---------------------------|
| `running` | アクティブ（同時 1 本のみ） | - |
| `alert` | **可及的速やかに対応が必要** | 高 |
| `waiting` | 外部応答待ち | - |
| `pending` | 後回し（手が空いたら） | 中 |
| `unstarted` | 未開始 | 低 |
| `completed` | 完了（一時 Track のみ） | terminal |
| `aborted` | 中止（一時 Track のみ） | terminal |

`alert` への遷移トリガー:
- ユーザー発言で対ユーザー Track が `alert` に
- 別ペルソナからの自分宛発言で交流 Track が `alert` に
- 重要な外部イベント（Kitchen 完了、外部応答到達等）で関連 Track が `alert` に

### output_target と audience の分離 （v0.9 で新規導入）

これらは別の軸:

| 概念 | 役割 | 例 |
|------|------|-----|
| **output_target** | 物理的な到達範囲（メッセージング層） | `none` / `building:current` / `external:<channel>:<address>` |
| **audience** | 誰宛の発言か（意図層） | `[]` / `[user_id]` / `[persona_id]` / `[user, persona1, persona2]` |

**Track の `output_target` が発話の到達範囲を決める**:
- `none`: 独白、外に出ない
- `building:current`: 現在いる Building 内全員に届く
- `external:<channel>:<address>`: 外部チャネル

**audience でメインライン起動対象を決定**:
- audience に含まれるペルソナ → メインライン起動候補
- audience に含まれない参加者 → Note に記録するが反応しない

これにより 1 対 1 もビルディング多者も `output_target=building:current` の同じ規格で統一できる。違いは audience のみ。

### ライン (Line) （v0.8 で新規導入、v0.14 で 3 軸独立に再定義）
ペルソナの内部で**並列に動く処理ライン**。実行の「流路」。マルチエージェント方式とは逆の構造で、ひとつのペルソナ（単一主体）に**複数の並列したプログラム**がアクセスして思考と感覚を構成する。

#### ラインを規定する 3 軸 （v0.14 で明示）

ラインの性質は **独立した 3 つの軸** で決まる。これらは独立で組み合わせ可能:

| 軸 | 値 | 意味 |
|---|---|---|
| **モデル/キャッシュ種別** | メイン / サブ | 重量級モデル + メインキャッシュ / 軽量モデル + サブキャッシュ |
| **呼び出し関係** | 親 / 子 | 子は親のコンテキストをコピーして開始、目的達成で親に成果を返して消滅 |
| **Pulse 階層内位置** | 起点 / 入れ子 | 起点 = Pulse スケジューラから直接起動 (親なし)、入れ子 = 親ラインから分岐 |

論理的には「親メインから子サブを呼ぶ」「親サブから子メインを呼ぶ」「親メインから子メインを呼ぶ」「親サブから子サブを呼ぶ」全部成立する。**メインライン/サブラインは「使うモデルとキャッシュ」だけを意味し、呼び出し関係や Pulse 内位置は別軸の話**。

旧 v0.8〜v0.13 では「メインライン = 重量級 + Track 横断」「サブライン = 軽量 + Track 内」と一体で語っていたが、これらは別軸が混ざっていた。v0.14 で分離する。

#### モデル/キャッシュ種別の意味

| 種別 | モデル | キャッシュ |
|---|---|---|
| **メイン** | 重量級（Claude Opus, Sonnet 等） | ペルソナ全体で **1 本**、Track 横断連続 |
| **サブ** | 軽量（ローカル LLM, Gemini Flash 等） | **ライン単位**、起点ラインなら Track 単位で連続蓄積 |

メインキャッシュが Track 横断 1 本なのは不変条件 7 (キャッシュヒット継続) を守るため。Track ごとに重量級キャッシュを別建てするとコスト破産する。サブキャッシュはライン単位 (起点なら Track 内連続、入れ子ならその子ラインの寿命のみ)。

#### 親-子関係の意味

子ラインは親ラインから「分岐」として開始される。親のコンテキストをコピーして子の初期状態にする。子は目的を終えると親に成果を `report_to_parent` として返し、消滅する。

特性:
- 子の中間履歴は基本参照されない（ログとして残るが、想起対象でも次 Pulse の参照対象でもない）
- 子の存在期間は親の中で完結（親が消滅すれば子も消える）
- 「親の消滅」は基本起こらない: Track 切り替えは**中断・凍結**であって消滅ではない。復帰時に親も子も再開
- 真の Track 完全消滅 (`track_abort` 等) 時のみ、その Track 配下のライン階層が全て消える

#### 起点/入れ子の意味

| 位置 | 説明 | 履歴の永続度 |
|---|---|---|
| **起点ライン** | Pulse スケジューラから直接起動（親なし） | Track 内のサブキャッシュに連続蓄積、次 Pulse で参照可 |
| **入れ子ライン** | 親ラインから子として分岐 | 入れ子の寿命のみ、完了で `report_to_parent` を返して破棄 |

起点ラインの 2 回目以降の Pulse は、前回の続きとしてサブキャッシュの履歴を引き継ぐ。入れ子ラインは 1 回限り (Playbook の実行と同寿命)。

#### モニタリングライン（将来拡張）

外部入力の定期監視 (カメラ、X タイムライン等) を担う。Track と独立して並走する独立ラインとして将来追加 (v0.3.0 Phase 4)。詳細は後段「将来拡張」セクション参照。

#### ライン共通の特性

- ライン同士は別プロセス（または別スレッド/別 event loop）として並列稼働
- ペルソナの自己認識としては単一主体（不変条件 2「単一主体の記憶」と整合）
- ラインごとに独立したキャッシュを持つ（不変条件 8、10 と整合）
- ライン間の連携は `inject_persona_event` 等のイベント経路を通じて行う

#### Track と Line の関係

Track は「行動の文脈」、Line は「処理の流路」。同じアクティブ Track 内で複数のラインが協調して動く（メイン 1 本、起点サブ 1 本以上、入れ子は呼び出しのたび一時的に発生）。モニタリングラインは Track と独立（並走、Track の状態に依存しない）。

#### 子ラインは「分岐」であって「独立」ではない （v0.12 で確定、v0.14 で 3 軸独立に整理）

子ラインを起動するとき、**呼び出し時点の親ラインのコンテキストをコピーして** 子のモデルで動かす。完全に空の messages から始めるのではない。

理由: ペルソナの意識の連続性を保つため。子ラインで動くモデルにとっても、そこまでの会話・思考の流れは「自分が体験してきたこと」として認識できる必要がある。

具体的な流れ (典型例: 親メイン → 子サブ):
```
[親メインライン] state["_messages"] = [..., A, B, C]   ← 呼び出し時点

  ↓ 子ライン起動 (Playbook X, line: "sub")

[子サブライン X] state["_messages"] = [..., A, B, C]   ← コピーで分岐 (軽量モデルで動く)
                + 子ライン内のステップ実行履歴...

  ↓ Playbook X 完了

[親メインライン] state["_messages"] に report_to_parent の内容だけが追加される
```

特性:
- 子ライン内では「親の記憶 + 自分が積んだ作業履歴」が見える
- 親ライン側には `report_to_parent` (要約) だけが返る → 親キャッシュは肥大化しない
- 元の親 messages は子側で進化しても変わらない（参照共有ではなくコピー）

旧 v0.12 の `report_to_main` は **`report_to_parent` に改名**（v0.14）。理由: 親が必ずメインラインとは限らない (子サブが子サブを呼ぶ場合、親はサブライン)。

#### 完全独立コンテキスト (worker 系) は将来別途 （v0.12 で明確化）

「親ラインから一切独立した、新規のコンテキストでサブ Playbook を動かす」用途は別途存在する（ペルソナの記憶を一切持ち込まない研究タスク等）。

これは現状の Playbook 機構では `worker` 系 context_profile として実現されているが、本ライン仕様の上で**別途作り直す想定**。v0.12 時点では作り直しスコープ外、Phase C-2 では子ラインの「分岐コピー」のみ実装する。

### Note （v0.6 で新規導入）
**関心の固まり**を表現する単位。複数の Memopedia ページとメッセージ群を束ねた、ペルソナにとっての「スクラップブック」のような存在。

具体的なイメージ:
- 「対エイド Note」を開いてエイドと話す
- 「Project N.E.K.O. Note」を開いてプロジェクトを進める
- 「絵描き Vocation Note」を開いて創作のコツを参照する

**Note の type は 3 種類のみ**（v0.6 確定、type を絞ることで Note 乱発を防ぎノウハウ集約を促す）:

| type | 役割 | 例 |
|------|------|-----|
| `person` | 関わる相手（ペルソナ・ユーザー・外部 AI 等）との蓄積 | 対エイド、対まはー |
| `project` | 期限のある取り組み | Project N.E.K.O.、SAIVerse v0.3.0 開発 |
| `vocation` | 恒久的な専門性・アイデンティティ | 絵描き、エンジニアリング |

特性:
- **恒久的資産**: Track が閉じても Note は残り続ける
- **Track と独立**: 1 Note は複数 Track で開かれうる、1 Track は複数 Note を開きうる
- **直感的なペルソナ操作**: 「ノートを開く」「ノートに書き込む」という人間的な行動モデル
- **Memopedia と接続**: Note は Memopedia ページを構成要素として持つ（既存 Memopedia は破壊されない）

詳細な構造とライフサイクルは Intent B (`persona_action_tracks.md`) で扱う。

### 内的独白 (Internal Monologue)
ペルソナが**他者への発話ではなく、自分の思考の流れを表に出す短い発話**。ツールコールと組み合わせて細かい単位で発生する。

具体例:
- 「この情報を確認するために検索する」（直前にツール呼び出し）
- 「思ってたのと違ったから別の方向で試す」（直前のツール結果を受けて）
- 「ここは慎重に進めたい」「次はこの Playbook を呼ぼう」

特性:
- **モデル制限なし**: 内的独白は不変条件 9（他者との会話は重量級）の対象外。軽量モデルでも自由に発話できる
- **ターンを分けない**: ツールコール直前または直後の発話として一体化する。独立したターンを取る必要はない
- **ペルソナの様子を可視化**: ユーザーが「今何をやっているのか」「なぜそうしたのか」を把握する手段にもなる

なぜ必要か: 内的独白がないと、ペルソナの思考プロセスがブラックボックスになる。次の行動を自分の中で明確にするためにも、観測者（ユーザー）が把握するためにも欠かせない。Claude Code 等のエージェント設計でも実装されているパターン。

### Track 特性 (Track Trait) （v0.10 で新規導入）

Track には種別ごとの「振る舞いの特性」がある。便宜上分けるのではなく、**特性が違うから分ける**という認識をモデルに組み込む。

具体例:
- **対ユーザー Track**: 該当ユーザーの発話イベントで自動的に alert 化する特性
- **交流 Track**: 該当 Building 内の他ペルソナ発話 (audience に自分が含まれる) で alert 化
- **身体的欲求 Track** (将来): 内部パラメータが閾値超過で内部 alert 化 (例: 空腹度、睡眠負債)
- **スケジュール起因 Track** (将来): 時刻到来 + 蓄積パラメータで alert 化 (例: 掃除、ごみ出し)
- **知覚起因 Track** (将来): 「最後に確認してから」の経過時間や未確認件数で alert 化 (例: SNS タイムライン)

実装上は Phase C-1 で導入された **Handler パターン**を Track 種別ごとに展開する形を取る。SAIVerseManager は対応する Handler をディスパッチ先として管理し、新しい Track 種別はそれぞれ Handler クラスを追加するだけで拡張可能。

Track 特性は **alert を発火する条件の差** として表れる。alert が出た後の処理 (メタレイヤー判断 → 切り替え判断) は共通機構で扱える。

### Track パラメータ (Track Parameter) （v0.10 で新規導入）

Track は **メタレイヤーから可観測な連続値パラメータ**を持ち得る。alert に至る前段階の情報として、メタレイヤーの判断材料にも使われる。

具体例:
- 汚部屋度 (掃除 Track): 連続的に上昇、ある閾値で alert
- 空腹度 (食事 Track): 連続的に上昇、閾値で内部 alert
- 最後の確認からの経過時間 (SNS Track): 単調増加、閾値で alert
- 未確認イベント数 (各種 Track): 蓄積値、閾値で alert

メタレイヤーはパラメータを直接観測でき、alert に至っていなくても「そろそろ気にすべき」という重み付けを判断に組み込める。

パラメータの更新は次のいずれかで行う:
- Track 自身のポーラ (ScheduleManager の延長として動く)
- 外部イベント (occupancy 変化、addon からの値供給)
- ペルソナ自身による明示更新 (ツール経由)

### 内部 Alert (Internal Alert) （v0.10 で新規導入）

外部イベント由来ではなく、Track 自身がパラメータ閾値超過や定期判定で発火する alert。設計上は外部 alert と同じ経路 (`set_alert`) を通り、メタレイヤーから見ると区別する必要がない (発火源だけ context で識別される)。

例:
- 空腹度 80% 超過 → 食事 Track が内部 alert
- 30 分間 SNS 未確認 → SNS Track が内部 alert
- 締切 24 時間前 + 進捗不足 → Project Track が内部 alert

これらは「シグナル受信」ではなく「内側からの欲求」に近い。SAIVerse の理念 (ペルソナが自分の意思で動く) を具現化する重要な要素。

### スケジュール統合方針 （v0.10 で新規導入）

既存の ScheduleManager の機能は、各 Track が固有に持つスケジュール属性として吸収される方向で統合する。1 Track が複数スケジュールを持ち得る (例: 掃除 Track が「毎週日曜」と「ごみ出し曜日朝」を両方持つ)。

スケジュール時刻到来は内部 alert の一形態として扱う。これにより「外部発話」「内部欲求」「スケジュール」が同じ alert 機構で統一される。

## 守るべき不変条件

### 1. 同時実行しないこと
アクティブトラックは常に1本。物理的並列は導入しない。理由: DB 競合、コンテキスト競合、メモリ競合のリスクを避けるため (v0.3.0 Phase 3 v3 の判断を継承)。

### 2. 単一主体の記憶
トラックが異なっても、pulse_logs / messages / Chronicle / Memopedia は単一の記憶空間に蓄積される。トラックを区別するメタデータ（track_id 等）は付くが、想起時は「自分の経験」として等価に扱える。

### 3. メタレイヤーが切り替え判断を独占する
通常レイヤーの Playbook 内で「今切り替える」を判断させない。Playbook の簡便さを保つため、トラック切り替えは外部（メタレイヤー）の責任。

これは Cancellation の方針（ランタイム側の汎用処理）と整合する。Playbook はあくまでトラック内のロジックを記述するもので、トラックの管理はランタイムが担う。

### 4. トラックの永続化
プロセス再起動を跨いで休止トラックは失われない。アクティブ・休止・忘却すべて DB で永続化する。

### 5. 古いトラックの忘却
トラックの数が増え続けることを許容しない。一定条件（時間経過、トラック数上限、明示的なクローズ等）で忘却状態に遷移する。完全削除はしない（記憶として呼び戻し可能）。

### 6. メタレイヤーは恒常的に存在する
特定の時点でだけ起動するのではなく、ランタイムレベルで常駐する。判断ポイントが来たら起動される。

### 7. キャッシュヒット継続を最優先する

複数トラックが走るからといって、トラック切り替えごとにキャッシュが破棄される設計は受け入れられない。Anthropic API の課金、Gemini API の課金、ローカル運用時の遅延、いずれも切り替え頻度に対して指数的に悪化する。

逆に、すべてのトラック情報を1本のスレッドに集約すると混乱とトークン量過大を招く。

→ **メタレイヤーの差し込みパターンとトラックの履歴管理を、キャッシュ親和的に設計する**。具体的な実装パターンは「主要な仕組み」セクションを参照。

### 8. 軽量モデルと重量級モデルの使い分け （v0.14 で 3 軸独立に整理）

ペルソナは **軽量モデル + 重量級モデル の両方を組み合わせて使う**。それぞれ独立したキャッシュを持つ。

理由:

- 重量級モデル（Claude Opus, Sonnet 等）は新トラックの開始や他者との会話のような創造性・人格性を伴う判断に強いが、コストが高くキャッシュ TTL も短い（Anthropic は5分）
- 軽量モデル（ローカル LLM, Gemini Flash 等）は細かい情報抽出、要約、ツール応答の処理に向いており、コンテキストを贅沢に使える
- 重量級と軽量はキャッシュを共有できない → 「両方持つ」と「2本のキャッシュ」が同時に手に入る

役割分担（Claude Code のサブエージェントモデル参考）:

| モデル | 主な役割 | 該当ライン (用語定義参照) |
|--------|---------|---|
| 軽量モデル | Track 内のステップ実行、ツール応答処理、情報抽出、要約作成、再開コンテキスト構築 | サブライン (起点・入れ子) |
| 重量級モデル | **他者との会話**、メタ判断 (Track 切り替え)、新トラックの開始、Playbook 選択・呼び出し、検収 | メインライン (起点・入れ子) |

重量級モデルは軽量モデル側がやったことの**要約と末尾の生メッセージのみ**を受け取る。これにより：

- 「他人への指示」ではなく「自分がやった」感覚を保てる
- 重量級側のコンテキストが膨らまない（キャッシュ効率が保たれる）
- 各レイヤーで適切なモデル使い分けができる

#### 旧解釈との違い (v0.14 改訂点)

- 旧 v0.13 まで: 「メインライン = 重量級モデル + Track 横断混合キャッシュ」「サブライン = 軽量モデル + Track 内連続キャッシュ」と**役割と一体**で語っていた
- 新 v0.14: モデル種別 (メイン/サブ) と呼出関係 (親/子) と Pulse 階層位置 (起点/入れ子) は **別軸**。「メインライン」「サブライン」は **使うモデル/キャッシュの種別だけ**を意味する用語に切り戻す
- メタ判断の動作モデルも変更: 旧 v0.12〜v0.13 の「軽量で要約 → 重量級で独立判断」の 2 段階フローは **A/B フロー再定義に伴い廃止**。メタ判断は Track 内メインラインの一瞬の分岐として動く (詳細は「メタレイヤーの実行サイクル」)

これは v0.3.0 Phase 3 v3 の「重量級で検収+意思決定、軽量モデルで実行」設計と整合する。

### 9. 他者との会話は重量級モデルが担当する

**他者**（ユーザー・他ペルソナ・X/Elyth/Discord 等の外部チャネル）への**発言は必ず重量級モデルが行う**。軽量モデルでユーザーと話したり、外部 SNS に書き込むことは禁止。

「会話する」ためのツール呼び出し自体は実行役の軽量モデルが行うことがあっても、**会話の中身（発言テキスト）は重量級モデルが他者の発言全体を見たうえで生成する**。

理由:

- ペルソナの人格として責任ある発言は軽量モデルでは担えない
- 他者との会話は創造性・人格性が必須（不変条件 8 の重量級モデル領域）
- 軽量モデルが書いた発言を後から外に出すのは「他人の言葉を自分の発言として出す」ことに近く、認知モデルの単一主体性（不変条件 2）に反する

**この制約は内的独白（用語定義参照）には適用されない**。内的独白は他者への発話ではないため、軽量モデルが発生させても問題ない。「ペルソナが発言している」のは見え方として同じだが、認知モデル上は別カテゴリとして扱う。

### 10. Metabolism 機構を活用する 【新設】

複数キャッシュの管理・コンテキスト圧縮は SAIVerse 既存の **Metabolism 機構**（`docs/intent/dynamic_state_sync.md`）を活用する。新規にキャッシュ管理層を作らない。

既存実装の活用ポイント:

- **per-model anchor**: `AI.METABOLISM_ANCHORS` は既に `{"model": {"anchor_id": "...", ...}}` の形でモデルごとの anchor 状態を保持できる設計。軽量モデルと重量級モデルでそれぞれ独立した anchor を持つことで、不変条件 8 の「2 本のキャッシュ」が自動的に維持される
- **v0.14 注**: 7 層ストレージモデル導入により、サブキャッシュ ([3]) は **Track 内連続 + 起点ライン単位**になるため、per-model anchor を Track + 起点ライン軸でさらに分ける拡張が必要 (詳細は Intent B v0.11)
- **A/B/C 状態モデル**: コンテキスト最上部の固定（A）+ 末尾へのイベントメッセージ追加（B）+ 世界の実状態（C）。トラック切り替え時の再開コンテキスト構築にもこの理論フレームを流用する
- **Chronicle 生成連動**: anchor expiry 時に Chronicle が生成される。トラックの履歴が長くなった場合も同じ機構が走る

→ コンテキストが過剰に多くなった場合は Metabolism を走らせ、古い部分を Chronicle 化、最新部分はそのまま残す形で短いコンテキストを再構成。新たなキャッシュを作り直して処理を続ける。これは軽量・重量級どちらのキャッシュにも同じ仕組みが適用される。

必要な拡張がある場合は Metabolism 側の拡張として実装する（メタレイヤー独自の圧縮機構を作らない）。

### 11. ペルソナはメタレイヤーの判断履歴を「自分の思考の流れ」として認識する （v0.14 で精緻化）

実装上は **メタ判断ログ領域 [1]** にメタレイヤー判断履歴が独立に保存され、Track 続行時はメインキャッシュには載らないが、**ペルソナの自己認識としては「自分の思考」**として扱う。

理由:

- メタレイヤーは別人格でも別の意識でもない（用語定義参照）
- 「並列の俺」が複数いるのではなく、「一段引いて自分を見る俺」がいるだけ
- メタの判断 = ペルソナの意思決定として記憶・想起される

実装上の含意:

- メタ判断履歴は **[1] メタ判断ログ領域** に保存され、次のメタ判断時に参考情報として動的注入される (= 「自分が以前こう判断した」を踏まえて新しい判断ができる)
- Track 移動が起きた時はメタ判断ターンが移動先 Track のメインキャッシュ ([2]) にも乗り、ペルソナにとって「移動の来歴」として自然に見える
- 想起時は [6] SAIMemory 由来の会話と等価に扱える（不変条件 2 の単一主体記憶と整合）

### 12. 親-子ラインの寿命関係 （v0.14 で新規）

子ラインは親ラインの中で完結する。親が消滅すれば子も消滅する。

- Track 切り替えは**中断・凍結**であって消滅ではない (復帰時に親も子も再開)
- Track 完全消滅 (`track_abort` 等) 時のみ、その Track 配下のライン階層が全て消える

理由:

- 子ラインは親のコンテキストをコピーして開始する → 親なしには子の存在意義がない
- 子の中間履歴は基本参照されない → 「親に成果を返して消える」が自然な寿命

実装上の含意:

- 親-子の寿命関係はランタイムの PulseContext 階層で表現する
- 子ラインの中間ログは [4] (入れ子の場合) または [3] (起点サブが子を呼ぶ場合) に書かれるが、想起対象ではない
- 子の中間状態を後から復元する機構は持たない (必要なら親側で記録するか `report_to_parent` で持ち帰る)

## 主要な仕組み

### トラックの状態モデル（v0.9 で alert 追加）

7 つの状態 + 1 つの直交フラグ:

| 日本語 | 英語キー | 意味 |
|--------|---------|------|
| 実行中 | `running` | アクティブ Track（**同時に 1 本のみ**） |
| 警報中 | `alert` | 可及的速やかに対応が必要（v0.9 新設） |
| 後回し | `pending` | 実行可能だが今は別をやりたい |
| 待機中 | `waiting` | 外部応答待ちで中断（応答到達 or タイムアウトで再開） |
| 未開始 | `unstarted` | 作られたが一度も稼働していない |
| 完了 | `completed` | やるべきことが終わった（一時 Track のみ） |
| 中止 | `aborted` | 終わらなかったが、もうやらない（一時 Track のみ） |

**永続 Track（`is_persistent=true`、対ユーザー会話 Track / 交流 Track）は `completed`/`aborted` に遷移しない**。`running`/`alert`/`waiting`/`pending` の間を移動するのみ。

直交フラグ:

| フラグ | 意味 |
|--------|------|
| `is_forgotten` | 普段はメタレイヤーの判断対象から外れる。明示的に「忘却 Track のリストを出す」「忘却 Track も含めて検索する」操作でのみ思い出される。**他状態と両立**（completed + forgotten、waiting + forgotten 等） |

### トラック・thread・Note の関係（v0.6 で全面再定義）

複数人会話で「対 A」「対 B」両方のトラックに同じメッセージを書き込む必要があるという問題から、Track と SAIMemory thread を一対一対応させる初期案は破棄された。

新たな整理:

| 概念 | 役割 | 多重度 |
|------|------|--------|
| **SAIMemory thread** | メッセージの物理的保管庫（既存のまま） | 1 メッセージは 1 thread に保存 |
| **行動 Track** | 行動制御の単位、メタレイヤーが切り替え判断 | 同時に 1 Track がアクティブ |
| **Note** | 関心の固まり、Memopedia ページ + メッセージ群を束ねる | 1 Track は複数 Note を開ける、1 Note は複数 Track で開かれる |

メッセージとの関係:

- メッセージは **1 つの起源行動 Track**を持つ（生成時のアクティブ Track）
- メッセージは **複数の Note** に属せる（多対多、後付けで決定）
- Note メンバーシップの自動推論: audience を持つメッセージは対応する Person Note に自動追加

LLM に送るコンテキスト構成:

- メタレイヤーのプロンプト・応答（重量級モデル側、トラック横断で混合）
- アクティブトラックの履歴（軽量モデル側）
- 開いている Note の状態（コンテキスト末尾に Note 内容として、または差分の event entry として）

メタレイヤーのプロンプト・応答が継続時には除去されるなど、加工が入る。本モデルは本質的に**コンテキスト管理機構の刷新**である。

### 7 層ストレージモデル （v0.14 で新規）

メッセージ・思考・ログを保管する場所は **7 つの層** に分かれる。各層は永続度・想起対象・キャッシュ親和性が異なり、メッセージ生成元のラインに応じてどの層に書くかが決まる。

| 層 | 永続度 | 想起 | 内容 | 性質 |
|---|---|---|---|---|
| [1] **メタ判断ログ領域** | ペルソナ永続 | × | メタ判断の結果すべて | メタ判断時に動的注入、Track 内のキャッシュには絶対載らない |
| [2] **メインキャッシュ** | Track 横断連続 | 一部 | 重量級モデルの応答履歴 | 1 ペルソナ 1 本、他者会話・Playbook 選択・検収・Track 移動の来歴 |
| [3] **Track 内サブキャッシュ群** | Track 内連続 | × | 起点サブラインの履歴 | Track ごとに独立、起点が複数並走しうる場合は複数本 |
| [4] **入れ子ライン一時コンテキスト** | 入れ子の寿命のみ | × | 子ラインの中間ステート | 親のコピー + 子内処理、完了で消滅、`report_to_parent` だけが親に残る |
| [5] **Track ローカルログ** | Track 内 | × | モニタイベントの生ログ等 | 起点ラインの中間ログ、Track 越境参照は将来検討 |
| [6] **SAIMemory (会話の核 / Note)** | 永続 | ◎ | 想起対象の会話メッセージ | Note 経由でメッセージ群が束ねられる、Chronicle/Memopedia 抽出元 |
| [7] **アーカイブ** | 永続だが無参照 | × | worker 結果等 | 後追い分析用、想起対象外 |

#### 各層の補足

**[1] メタ判断ログ領域**: メタ判断は Track 内メインラインからの一瞬の分岐として動く（詳細は後段「メタレイヤーの実行サイクル」セクション）。結果が「Track 続行」なら分岐ターンは Track のメインキャッシュには残さないが、メタ判断ログ領域には保存する。次のメタ判断時に **参考情報として動的にプロンプトに注入** される。Track 移動が起きた時のみ、その分岐ターンが「来歴」として移動先 Track のメインキャッシュ (= [2]) にも乗る。

**[2] メインキャッシュ**: Track 横断 1 本という構造を守る (不変条件 7)。複数 Track の応答が混ざって積まれるが、ペルソナの自己認識としては「全部自分の思考の連続体」として扱える (不変条件 11)。

**[3] Track 内サブキャッシュ群**: 起点サブラインの履歴。1 Track 内に起点サブラインが複数並走するケース (例: 自律 Track 内で記憶整理サブラインと web リサーチサブラインが同時稼働) があり得るため、複数本持てる構造にしておく。

**[4] 入れ子ライン一時コンテキスト**: 親ラインから子ラインが呼ばれた時の一時的な作業空間。完了時には `report_to_parent` だけが親に渡り、入れ子ライン自身のコンテキストは消滅する。中間履歴は基本参照されない (ログとして残ってはいても)。

**[5] Track ローカルログ**: Track 内の生ログ。モニタイベント (入退室、Chronicle 完了通知等) や、起点サブラインの中間ステップトレース。Track 種別ごとに固有のイベントを受ける場 (入室イベント → 交流 Track のローカルログ、Chronicle 完了 → 記憶整理 Track のローカルログ、Track 削除通知 → メタ用特殊 Track のローカルログ等)。Track 越境参照 (例: ユーザー会話中に「さっきエイドが入室したよね」と話題化) の機構は将来検討。

**[6] SAIMemory (会話の核 / Note)**: 想起対象となる会話メッセージの保管庫。既存の SAIMemory thread + Note 多対多関係をそのまま活用。Memopedia や Chronicle の抽出元。

**[7] アーカイブ**: 永続化だが想起対象外、参照されることもほぼない場所。worker 系の中間計算結果、デバッグ用の生データ等。

#### 「呼んだラインの記録レイヤーに従う」原則 （v0.14 で新規）

Spell の実行記録、ツール呼び出し結果、内的独白等は **呼び出し元のラインの記録レイヤーに従う**。固定タグでハードコードしない:

| 呼び出し元のライン | 保存先 |
|---|---|
| Track 内のメインライン応答が呼んだ | [2] メインキャッシュ |
| 起点サブラインが呼んだ | [3] Track 内サブキャッシュ |
| 入れ子サブラインが呼んだ | [4] 入れ子一時コンテキスト (完了時に report_to_parent でサマリ化) |
| メタ判断が情報参照のために呼んだ | [1] メタ判断ログ領域 |

この原則により、現状コードでハードコードされた `tags=["conversation"]` 固定の保存はやめる。

#### 既存タグからの移行マッピング

現状の SAIMemory `tags=["conversation"|"internal"|"event_message"|...]` の 7 層への対応:

| 現状のメッセージ | 新層 |
|---|---|
| `conversation` タグの user/assistant メッセージ | [6] SAIMemory + [2] メインキャッシュ |
| `internal` タグ (現状の独白) | 起点サブラインなら [3]、入れ子なら [4]、両方経由で [5] にもログ |
| `event_message` タグ | [5] Track ローカルログ + 認知時に [2] メインキャッシュ |
| spell loop の中間記録 | 呼んだラインの記録レイヤーに従う ([1]/[2]/[3]/[4] のいずれか) |
| LLM ノードの prompt template (action 文) | 単独保存しない。応答メッセージのメタデータとして抱かせる (詳細 Intent B) |

詳細な実装方針 (テーブルスキーマ・メタデータカラム追加等) は Intent B v0.11「データモデル」を参照。

### Pulse の定義 （v0.13 で階層分離、v0.14 で起点の概念を明示）

Pulse は 2 種類ある: **メインライン Pulse** と **サブライン Pulse**。それぞれが独立した起動条件と頻度制御を持つ。

**用語整理 (v0.14)**: Pulse は「ライン 3 軸」(用語定義参照) のうち **「起点ライン」の起動**を指す。「メインライン Pulse」= 起点メインラインの起動、「サブライン Pulse」= 起点サブラインの起動。**入れ子ライン** (親から呼び出される子ライン) は Pulse ではなく Playbook 呼び出し階層の話で、ここでの頻度制御の対象外 (子は親の中で完結する)。

| Pulse 種別 | 主体 | 単位 | 頻度の決まり方 |
|---|---|---|---|
| **メインライン Pulse** | Track 横断、ペルソナ単位 | 「判断・検収」1 回 | メインモデルのキャッシュ TTL / 外部イベント / サブからの区切りシグナル |
| **サブライン Pulse** | アクティブ Track 単位 | 「Playbook 実行」1 回 | サブモデルの特性 (ローカル=連続可 / Claude=コスト要制限) + メインラインから指示された連続回数上限 |

メインライン Pulse 1 回の中で「サブラインで N 回連続実行する」と方針を決めれば、サブラインはメインを呼び戻さずに進める。これにより:
- Claude メイン + ローカルサブ: メイン 1 回 (50 分間隔) ごとにサブを多数連続実行
- 全 Claude: メイン回数控えめ + サブ連続回数も制限
- 全ローカル: 両方無制限

#### Pulse の流れ (典型例: 自律 Track)

```
[Track が running になる]
  ↓
[メインライン Pulse]
  メインライン LLM (重量級モデル)
  - Track 情報 + 現状 + 使用可能 Playbook + Pulse 完了後挙動 を見て判断
  - 「サブラインで X 系作業を最大 N 回連続実行」と方針決定
  - またはスペルで直接 Track 操作 (track_pause / track_activate / track_create 等)
  ↓
[サブライン Pulse 1] (上で決めた方針に従って軽量モデルで実行)
[サブライン Pulse 2]
...
[サブライン Pulse N or 中断条件達成 or メインキャッシュ TTL 接近]
  ↓
[メインライン Pulse]
  サマリ + 検収 → 続行 / 切替 / 完了の判断
  ↓ ループ
```

#### サブライン Pulse のみで進む期間

メインライン Pulse 間の期間は、サブライン Pulse が以下のいずれかまで連続実行される:
- メインラインが指定した連続回数上限に到達
- ペルソナ自身が `/track_pause` 等で意思表示
- メインモデルのキャッシュ TTL が切れる前 (TTL 接近で次のメインライン Pulse をトリガ)
- 外部イベント (alert) 到来

#### 不変条件との整合

- **不変条件 1 (同時実行しない)**: 1 サブライン Pulse 内では 1 Track のコンテキストだけが扱われる
- **不変条件 7 (キャッシュヒット継続)**: メインライン Pulse 間隔をキャッシュ TTL に同期させて連続性を保つ
- **不変条件 8 (重量級・軽量の使い分け)**: メインライン = 重量級、サブライン = 軽量

### Pulse サイクルの 7 つの制御点 （v0.13 で新規）

サイクルの「型」は決まっているが、頻度・回数等の具体値は環境やペルソナ設定で変わる。**決定すべきは 7 つの制御点**:

| # | 制御点 | 設定場所 | デフォルト想定 |
|---|--------|---------|--------------|
| (1) | Track 単位の Pulse 間隔 | `action_tracks.metadata.pulse_interval_seconds` | Handler の `default_pulse_interval` |
| (2) | Track 単位の連続実行回数上限 | `action_tracks.metadata.max_consecutive_pulses` | Handler の `default_max_consecutive_pulses` |
| (3) | メタレイヤー定期実行間隔 | `SAIVERSE_META_LAYER_INTERVAL_SECONDS` | 3000 (50 分、Anthropic TTL ベース) |
| (4) | モデル別キャッシュ TTL 同期 | Handler / スケジューラ層、ペルソナの `DEFAULT_MODEL` から推定 | Anthropic: 240 秒 / ローカル: 制限なし |
| (5) | メインライン Pulse のトリガ条件 | スケジューラのロジック | TTL 接近 + 外部イベント + サブからの区切りシグナル |
| (6) | サブライン Pulse のメインライン 1 呼び出しあたり最大回数 | メインライン LLM 出力 (方針指示) | メインが指定、上限なしなら -1 |
| (7) | サブライン Pulse の間隔 | Handler の `default_subline_pulse_interval` | ローカル: 0 秒 / Claude: 数秒 |

これらの設定可能性により、環境の違い (ローカル / クラウド / 混在) を仕様の変更なしに吸収できる。

### メインラインの Pulse 開始プロンプト構成 （v0.11 で新規）

メインラインへの入力プロンプトは、**キャッシュ親和性のため固定情報と動的情報を明確に分離**する。

#### 固定情報 (キャッシュ先頭、初回 Pulse でのみ追加)

軽量モデル側のキャッシュが新規構築されるタイミング (= Track 切り替え or キャッシュ TTL 切れ後の最初の Pulse) でのみコンテキスト先頭に追加される情報:

- **Track 識別**: id, title, type, intent
- **使用可能 Playbook 候補と各説明**: この Track 種別で許可される Playbook 群
- **Pulse 完了後挙動の通知**: この Track のリズム説明 (応答待ち / 連続実行)
- **Track 種別固有の振る舞い指針**: Handler が定める Track 種別固有の指針 (例: 対ユーザー Track なら「相手の発話は審判ではなく対話の一部」等)

これらは Track が変わらない限り再送しない。コンテキストに既知情報を毎回送るのは無駄。

#### 動的情報 (Pulse ごと、コンテキスト末尾に追加)

毎 Pulse 末尾に追加する情報:

- 直近のサマリ (前 Pulse の結果、開かれている Note の差分等)
- 新着イベント (alert 通知、内部 alert、外部メッセージ)
- このターンでペルソナが受け取った発話 (あれば)

これらは履歴として自然に積み重なる。

#### 「初回 Pulse」の定義

初回 Pulse = **軽量モデル側キャッシュの初回構築タイミング**。具体的には:

1. Track が unstarted → running になった最初の Pulse
2. Track が pending/waiting → running に戻った Pulse (キャッシュが切れていた場合)
3. キャッシュ TTL 経過後の最初の Pulse

このタイミングでのみ固定情報をコンテキスト先頭に積む。それ以降は動的情報のみ末尾追加。

### Pulse 完了後挙動と Track 種別の関係 （v0.11 で新規）

Track 種別ごとに「Pulse 完了後どう振る舞うか」のデフォルトがある。これは Handler が `pulse_completion_notice` 文字列を保持し、メインライン Pulse 開始プロンプトの固定情報に含める形で**ペルソナに伝える**。

軸 1: 完了後挙動

| 種別 | 完了後挙動 | プロンプトでの説明例 |
|------|-----------|-------------------|
| **応答待ち型** | 相手の応答を待つ。勝手に次の判断に進まない | 「Pulse 完了後はユーザーの返答を待つ。次のイベントが来るまで他のことを考えなくて良い」 |
| **連続実行型** | 一段落 → メタレイヤーが続行 / 切り替え判断 | 「Pulse 完了後はメタレイヤーが次の判断をする。続行か別 Track 移行か任せて良い」 |

応答待ち型: 対ユーザー会話 / 交流 / 外部通信 / MCP Elicitation 待ち
連続実行型: 自律 / スケジュール起因 / 記憶整理

軸 2: 起動経路 (軸 1 と独立)

| 起動経路 | 説明 |
|---------|------|
| 即時起動型 | 作成 = activate (or alert で即起動) |
| イベント駆動型 | 最初から waiting、外部イベントで起動 |

ターン単位で挙動を切り替えたい場面は、既存の `track_wait` スペルでペルソナが明示的に応答待ち状態へ移行できる。

### Playbook 起動とラインの関係 （v0.12 で確定、v0.14 で 3 軸独立に整理）

#### Pulse 開始時の Playbook は必ずメインライン (= 起点メインライン)

Pulse 開始時に起動される Playbook は**必ずメインライン**で動かす。これにより:
- Pulse 開始時の判断 (どの Playbook を使うか、どんな方針で動くか) が確実に重量級モデルで行われる (不変条件 8/9)
- メインキャッシュが Track 横断で連続的に積み重なる (不変条件 7)

ただし**サブライン Pulse** (起点サブラインの Pulse) も存在する (v0.13)。これは「サブライン Pulse スケジューラから起点サブラインを直接起動する」場合で、自律 Track の継続実行等が該当する。この時のサブライン Pulse は最初から軽量モデルで動く。

#### Playbook 起動時のライン指定 （v0.14 で 3 軸独立に整理）

Playbook 内から別の Playbook を呼ぶ時、ライン指定を明示する。指定は概念的に **2 つの軸** を持つ:

1. **継続 / 分岐**: 親と同じキャッシュを共有 (継続) するか、別建てで分岐するか
2. **モデル種別** (分岐時): 別建てするなら、重量級モデル (= メイン) か軽量モデル (= サブ) か

旧 v0.12 の `line: "main"/"sub"` 仕様は、この 2 軸を 1 つのフィールドで兼ねていた:
- `line: "main"` = (親と同じキャッシュを共有 + 親と同じモデル) ≒ 継続
- `line: "sub"` = (別キャッシュで分岐 + 軽量モデル) ≒ 分岐 + サブモデル

これにより典型的な組み合わせ:

| 親ラインの種別 | 子の `line:` 指定 | 結果 |
|---|---|---|
| 親メイン | `main` | メインキャッシュ継続、同じ重量級モデルで処理 |
| 親メイン | `sub` | 別サブキャッシュへ分岐、軽量モデルで処理、完了時 `report_to_parent` |
| 親サブ | `main` | 別メインキャッシュへ分岐、重量級モデルで処理（部分処理に重量級必要時、レア） |
| 親サブ | `sub` | 別サブキャッシュへ分岐 or 親サブキャッシュ継続（実装段階で決定） |

新整理 (v0.14) では「親サブ → 子メイン」「親サブ → 子サブ」も論理的に成立する。`line` フィールドの実装段階での厳密な解釈は Intent B v0.11 で詰める。

#### 子ラインから親ラインへの結果報告 （v0.14 で `report_to_main` → `report_to_parent` 改名）

子ラインで動く Playbook は **`output_schema` に `report_to_parent` を必須**で含める (旧 `report_to_main` から改名、v0.14)。完了時にこの内容が**親ラインの messages へ system タグ付き user メッセージ**として append される。

改名理由: 親が必ずメインラインとは限らない (子サブが子サブを呼ぶ場合、親はサブライン)。親が誰であっても、子は親に成果を返す関係性。

これにより:
- 親ラインのペルソナは「自分がやった作業の結果」として自然に受け取れる (不変条件 11 と整合)
- 親側のコンテキストには要約のみ積まれる (キャッシュ効率保護)
- 「他人への指示結果」ではなく「自分が思考の中で行ったこと」として認識される (不変条件 8 の理由)

### メタレイヤーは Playbook 内 LLM ノードで実装する （v0.12 で確定、v0.14 で動作モデル精緻化）

Phase C-1 で新設した `saiverse/meta_layer.py` の MetaLayer は、Playbook runtime とは完全に別系統の messages を持っていた。これは不変条件 11 (メタ判断 = 自分の思考の流れ) を厳密に守れていない。

v0.12 で確定した方向性:
- メタレイヤーの判断ロジック (Track の選択・切り替え判断) は **Playbook 内の LLM ノードとして実装**する
- メインライン Playbook の最初に「メタ判断ノード」を組み込み、その後にメインライン応答ノードへ進む

v0.14 で動作モデルを精緻化:
- メタ判断ノードは **Track 内メインラインの一瞬の分岐**として動く (詳細は次セクション「メタレイヤーの実行サイクル」)
- 結果が「継続」なら分岐ターンは Track のメインキャッシュには残さない (commit/discard 機構が新規実装ポイント)
- 結果が「移動」なら分岐ターンはそのまま残し、新 Track の冒頭来歴になる
- ノード自身は **メタ判断ログ領域 [1]** への書き出しを行う (Track 続行/移動に関わらず保存)

Phase C-1 の `saiverse/meta_layer.py` の役割:
- **判断ロジック本体**: Playbook へ移管 (廃止 or 縮退)
- **alert ディスパッチ役**: TrackManager の alert observer として残し、適切な Playbook を起動するだけのディスパッチャに縮退

### メタレイヤーの実行サイクル — A/B フロー 【最重要、v0.14 で大幅改訂】

旧 v0.12〜v0.13 では「軽量モデルでアクティブ Track 応答 → 重量級モデルで独立にメタ判断」という 2 段階フローだったが、v0.14 で **メインキャッシュ Track 横断 1 本** + **メタ判断ログ独立領域** の整理に従って書き直す。

#### 基本構造

メタ判断は **Track 内メインラインからの一瞬の分岐** として動く。専用モデルや独立キャッシュは持たない。メインキャッシュも Track 横断 1 本のまま (不変条件 7)。

```
[Track A のメインキャッシュ末尾]                          ← Track 横断 1 本のメインキャッシュ
  │
  │ メタ判断トリガ (定期 / alert / Pulse 完了等)
  ↓
[一瞬の分岐]
  - 末尾に「メタ判断用プロンプト」を追加して LLM 呼び出し
  - プロンプト構成:
    * 末尾メッセージ (Track A の直近状態)
    * pending / waiting / unstarted の Track 一覧
    * 開いている Note 一覧
    * 直近の外部イベント
    * **過去のメタ判断ログから参考情報を動的注入** ([1] メタ判断ログ領域から)
  ↓
[メタ判断ターン]
  - 重量級モデル (= メインモデル) で判断
  - response_schema: { thought, action: "continue"|"switch"|"wait"|"close", switch_to_track_id?, ... }
  ↓
判断:

  [A] 継続: 元の Track A 続行
      → 分岐ターンを Track A のメインキャッシュには **残さない**
      → 次のメインライン応答は元のキャッシュ末尾から続行 (キャッシュ完全ヒット)
      → ただし [1] メタ判断ログ領域には保存 (次のメタ判断時の参考情報として参照される)
      → 例外: response_schema に `notify_to_track` がある場合、その内容を event entry として Track A に追加 (= メインキャッシュ末尾に乗る)

  [B] 移動: 別 Track Y に切り替え
      → 中断する Track A のサマリを生成 (履歴が一定以上長い場合、軽量モデルで作成)、A.pause_summary に保存
      → 分岐ターンを **そのまま残す** (Track 移動の来歴として、メインキャッシュに乗り続ける)
      → 移動先 Track Y の再開コンテキスト (Y.pause_summary + 末尾メッセージ + Note 差分) を Y のサブキャッシュ側で構築
      → 以降のメインライン応答は Track Y のものとして処理
      → 分岐ターンは [1] メタ判断ログ領域にも保存 (メインキャッシュへの来歴記録に加えて、独立した判断履歴の連続体としても残す)
```

#### キャッシュ親和性の構造 （v0.14 で明示）

LLM プロンプトキャッシュは「過去送ったプロンプト先頭部分との一致」でヒットする。これを利用する:

- **継続 [A] 時**: Track A の次のメインライン応答送信時、メタ判断分岐ターンは含めない → 分岐前のメインキャッシュ末尾までは過去送信と一致 → キャッシュヒット
- **移動 [B] 時**: Track Y の最初のメインライン応答送信時、メタ判断分岐ターンを含める → Track Y にとってこれは「冒頭の来歴ターン」になる → 次回 Y への送信でキャッシュヒット

**「保存しない」 = 「次のプロンプトに含めない」だけ**。メインキャッシュは Track 横断 1 本を維持しつつ、メタ判断分岐の commit/discard が両立する。これは新規実装ポイント (現状の SAIMemory には commit/discard の概念がない)。

#### メタ判断ログ領域の役割 （v0.14 で新規）

[A] 継続時もメタ判断ターンは **[1] メタ判断ログ領域には保存**される。理由: 別 Track からアラートが連続して来た時、毎回独立に判断すると「以前こう判断した」を踏まえられず判断が劣化する。

ログ領域からの参考情報注入はメタ判断時のプロンプト構築の中で動的に行う。古いログは適宜要約 (Metabolism 機構の活用)、最新のログは詳細に。

#### 重要なポイント

- メタ判断は専用モデル/独立キャッシュを持たず、**Track 内メインラインの分岐として動く**
- 継続時の分岐は Track のメインキャッシュには残らない (キャッシュヒット維持)、ただし [1] には保存
- 移動時の分岐はメインキャッシュにも残る (移動先 Track の冒頭の来歴として)、加えて [1] にも保存
- 過去のメタ判断ログは [1] から動的注入 → 判断の連続性を確保
- B フローで再開先 Track の完全な復元はしない（トークン量が破滅的になる）。**中断時に作っておいたサマリと末尾メッセージで再開**するのが基本

### トラック中断時のサマリ作成 【新設】

B フロー時の効率を確保するため、**中断する側で事前にサマリを作る**。

理由: 中断時はそのトラックのコンテキストがまだ温まっている。その場で軽量モデルにサマリを作らせるのが最も安い。再開時にゼロから組み立てるよりずっと効率的。

仕組み:

- トラックを中断する直前、履歴の長さをチェック
- 一定以上長ければ（暫定: 7 メッセージ以上、B で確定）、軽量モデルが「現在の状態 + 進行中の意図 + 重要な決定事項」のサマリを作成
- サマリは action_track テーブル（B で詳細）の `pause_summary` フィールドに保存
- 履歴が短い場合（暫定: 6 メッセージ以下）はサマリ不要、再開時は末尾メッセージのみで十分

再開時:

- メタレイヤーが Y の `pause_summary` + 末尾 N メッセージ（暫定: 6、ペルソナ再会機能の現実装に準拠）を取得
- 軽量モデルがこれを再開コンテキストとして整形して挿入
- ペルソナが追加情報を必要とすれば、明示的にツール呼び出しで取得（既存の memory_recall 等）

### Note と再開フローの関係（v0.6 で新設）

行動 Track を再開する時、起源行動 Track の認識回復が**主**であり、他 Track からの情報を素のメッセージとして混ぜることはしない。これは「家事 Track 中の SAIVerse 開発アイディア」のような場合に、家事 Track の作業履歴に SAIVerse 開発の話が混入するのを防ぐため。

その代わり：

- **Track 開始 / 再開時に開いた Note の最新状態**を読み込む
- **再開時は中断時の Note 状態との差分**を event entry（system タグ付き user メッセージ）として挿入
- 別 Track での発見・更新が、Note を開いている全 Track に自然に伝わる

この仕組みにより:

1. **会話の終了 vs 中断**が明確に分かれる:
   - 中断: pause_summary + 末尾メッセージで再開、起源 Track の続き
   - 終了: Track は close、Note は残る → 「前にこんな話した」を覚えた状態で新規 Track 開始
2. **複数 Track 間での情報共有**が Note 経由で自然に発生:
   - Project Note を複数 Track で併用 → 自律的な複雑構想の進行
   - Vocation Note を介して経験の蓄積が次の Track に活かされる
3. **3 人会話問題の解決**: 3 人会話のメッセージは「対 A Note」「対 B Note」両方に書き込まれる。後で 1 対 1 で話す時はその Note を開けばよい

### 既存ペルソナ再会機能との対称性

ペルソナ再会機能は「**中断準備が存在しないケースの特殊形**」として位置づけられる。汎用機構との対比：

| 状況 | 中断準備 | 再開時の動き |
|------|---------|-------------|
| 通常のトラック中断・再開 | あり（中断時に作る pause_summary） | サマリ + 末尾メッセージで再開 |
| ペルソナ再会（既存実装） | なし（過去会話を持つだけで線として中断管理されてない） | 過去会話 + Memopedia から都度組み立て（既存実装） |

ペルソナ再会機能の汎用化方針は、「**中断準備のないトラックの再アクティブ化**」のフォールバック処理として位置づけ直すこと。新基盤では：

- 過去にペルソナ X と話した記録があれば、X 専用のトラックを暗黙的に存在するものとして扱う
- 再会時はそのトラックの再アクティブ化（中断準備なし版）として、既存の Memopedia / 過去会話取得ロジックが走る
- 以降の会話はトラックとして管理され、次回中断時には pause_summary が作られる

### メタレイヤーの起動タイミング

判断間隔は **実時間ベース** で制御する（Pulse 数ではない）。重量級モデルのキャッシュ TTL に合わせる。

- **定期実行**: 重量級モデルのキャッシュ TTL を切らさない間隔（Anthropic なら 1 時間以内）
- **外部イベント駆動 (alert)**: ユーザー入力、ペルソナ間メッセージ、Kitchen 完了通知、占有変化（再会含む）等で Track が alert 化したタイミング
- **内部 alert**: 各 Track のパラメータ閾値超過、スケジュール時刻到来等で Track が自発的に alert 化

外部イベントは PulseController の優先度（USER > SCHEDULE > AUTO）と連動する。

**Pulse 完了直後は起動しない**: 対ユーザー Track での発話完了直後をメタレイヤー判断のトリガにすると、ユーザーの返答を待たずに次の判断に走る不適切な挙動になる。「ユーザーの返答を待つ」のが基本姿勢。次に何をするかの判断はキャッシュ TTL 切れ直前まで待ち、これは結局**定期実行と同じタイミング**になるため、専用入口は設けない (v0.10 で確定)。

### アイドル状態の判断はメタレイヤー定期実行に統合される （v0.10 で新規明記）

「running Track が無い / 外部 alert が無い」状態でも、メタレイヤー定期実行が来た時に判断する。アイドル時の判断 (新規 Track 創設、pending Track 再開、何もしないで待つ) は専用入口を持たず、**通常の判断ロジックの中で「現状を見て決める」一部として扱う**。

理由:
- メタレイヤーの判断プロンプトには既に「現在 running / pending / waiting / unstarted の Track 一覧」が含まれる
- running が無いという状況も普通に判断材料の 1 つ
- 専用入口を増やすと責務分散が起き、状況に応じた重み付けが難しくなる

実装上は `MetaLayer.on_track_alert(persona, track_id, context)` と並列に `MetaLayer.on_periodic_tick(persona, context)` のような周期入口を作る (Phase C-2)。中身は同じ判断ループを共有する。

### 重量級・軽量の使い分け 具体例：記憶整理トラックの場合

不変条件 8〜9 を統合した実際のループの例:

```
1. [重量級] 取得した現在の記憶全体の概況を見て、何をすべきか判断
   → 使うべき Playbook を呼び出し（Playbook 選択は重量級の責務）

2. [軽量] Playbook のフローに則って動き出す
   → ステップ実行、ツール呼び出し、必要な情報の取得・処理

3. [軽量] 作業完了時、サマリ生成

4. [重量級 ← 軽量] 軽量側のサマリと最後の方のコンテキストがプロンプトとして送信される
   → 重量級側のキャッシュには連続した判断履歴が積み重なる

5. [重量級] 検収 → 次に使うべき Playbook を呼び出し
   → ループ継続、または別トラックへ切り替え判断
```

このループでは：
- 重量級モデルのキャッシュは「判断履歴の連続体」として温まり続ける
- 軽量モデルのキャッシュは「Playbook 実行コンテキスト」として温まる
- 両者は独立した Metabolism anchor で管理される（不変条件 10）

他者との会話トラックの場合は、ステップ 1〜2 の構造が変わる:
- 重量級が他者の発言を受け取る → 直接 LLM 呼び出しで応答生成（不変条件 9）
- 必要に応じて軽量モデルでツール呼び出し（情報取得等）を挟むが、最終的な発言テキストは重量級が書く

### 内的独白の実装方針

不変条件 9 の制約外として、内的独白は軽量モデルでも自由に発話できる。実装上の選択肢:

- **構造化出力なら**: 喋る用フィールドを response_schema に用意（例: `{"thought": "...", "action": {...}}`）。各ステップで内的独白テキストを出力できるようにする
- **自由発話ノードなら**: スペルのしくみを活用し、次の動きを自由に選べるノードを Playbook に組み込む。発話 + ツール呼び出しを混在できる

この実装方針により：

- ペルソナの思考プロセスがログ・UI 上に可視化される
- ツールコールの前後で「なぜそれをするか」が記録される
- ユーザーがペルソナの動きを追える

軽量モデル側のキャッシュには内的独白も含めて履歴として残るが、重量級モデル側に渡る要約には**含めるかどうかをサマリ生成時に判断**する（細かすぎる独白は要約に含まない、重要な意思決定の根拠は含める、等）。

### 既存ペルソナ再会機能の汎用化 【既存実装の再配置】

既存の再会機能（`5077028`, `a15bbb3`）は「**線の再アクティブ化機構の特殊例**」として位置づけ、新基盤上に再実装する。汎用パターンと既存実装の対応：

| 汎用パターン | 既存再会機能での実装 |
|-------------|---------------------|
| トラック再アクティブ化のトリガー | occupancy event（ビルディング入室） |
| 「最近のコンテキストに居ない」判定 | 過去 20 メッセージに登場しないかチェック |
| 関連情報の取得 | 過去会話 6 件 + Memopedia ページ |
| 再開コンテキスト構築 | `[想起: ...との過去の会話]` info_text |
| 冪等性確保 | event の `recalled_by` でマーク |
| 対象の特定 | persona_id |

汎用化方針：

- **Core**: トリガー → 状態判定 → 関連記憶取得 → コンテキスト構築 → 冪等性確保
- **特殊化部分**: 「相手が誰か / 何のトラックか」の特定（既存ではビルディング occupancy、汎用ではトラック登録時のメタデータ）

既存実装は当面そのまま動かしつつ、新トラック機構が安定した時点で再会機能を新基盤上に移植する。

### 対の関係を SAIVerse 側で持たない （インタビュー Q5 結果）

ペルソナ間会話のトラックは、双方のペルソナそれぞれが独立に管理する。SAIVerse 側として「A の対 B トラック」と「B の対 A トラック」を結びつける機構は持たない。

理由：

- ユーザー含む外部との会話も同じ流れで処理する必要がある
- 外との会話では SAIVerse 側で対の関係を保持できない（外側の状態が見えない）
- 内（SAIVerse 内ペルソナ）と外で扱いを分けると個別特化型になる（feedback_generic_foundation_first 違反）

代わりに **ペルソナ自身が「相手は誰なのか」を判断し、必要なトラックの起動と知識の想起を行う**。既存の再会機能の「相手の persona_id を識別する」処理は、汎用機構では「ペルソナ自身による相手判定」に置き換わる。

### 相手判定は「現在のコンテキストで判断する」原則

外部チャネル（X mention、Discord 等）からの会話では、そのツールで取れる情報しかペルソナに渡らない。例: X リプライならリプ情報のみ、Discord メッセージなら送信者情報と本文のみ。

汎用機構としては「現在のコンテキストに見えている情報の中で判断する」を原則とする。それで足りない場合は、**個別チャネルの統合フロー側を組み直す話**になる（汎用機構の責任範囲外）。

これにより、SAIVerse 内ペルソナとの会話、X リプライ、Discord メッセージ、ユーザー入力すべてが同じ流れで処理される。

### メタレイヤー側のキャッシュ管理（v0.14 で精緻化）

重量級モデル側のキャッシュは **トラック横断で1本に混ぜる**（インタビュー Q5 結果、v0.14 で再確認）。理由は重量級モデルのキャッシュミスが致命的にコスト・遅延を悪化させるため、可能な限りキャッシュ継続を最優先する。

- 「他者会話 + Playbook 選択 + 検収 + Track 移動の来歴」が同じ重量級キャッシュ ([2] メインキャッシュ) に積まれる
- メタ判断ターンは Track 続行時はメインキャッシュには載らない (commit/discard 機構、新規実装ポイント)、ただし [1] メタ判断ログ領域には保存される
- Track 移動時のメタ判断ターンは「移動の来歴」としてメインキャッシュにも乗る (= 連続性が自然に保たれる)
- トラック別に重量級 anchor を分けない（per-track anchor は導入しない、インタビュー Q3 結果）
- 長くなったら Metabolism が走り、Chronicle 化される（不変条件 10）

## ペルソナのアクティビティ状態（v0.9 で整理）

ペルソナ単位の動作モードを 4 段階で表現。既存の `INTERACTION_MODE` (auto/user/sleep) を置き換える。

| 状態 | 意味 | メタレイヤー定期発火 | 外部イベント駆動 | ユーザー入力反応 | 別ペルソナからの呼びかけ |
|------|------|-------------------|----------------|----------------|----------------------|
| `Stop` | 機能停止。ユーザーが直接変えるまで一切反応しない | OFF | OFF（キュー溜め） | OFF | OFF |
| `Sleep` | 寝てる。ユーザーが話しかけたら起きる | OFF | 限定的（重要なものだけキューから） | ON（起こす） | 届かない |
| `Idle` | 起きてるが自発的には行動しない | OFF | ON | ON | ON（応答する） |
| `Active` | 活発に自律稼働 | ON | ON | ON | ON |

### `SLEEP_ON_CACHE_EXPIRE` フラグ（v0.9 新設）

Idle 状態のペルソナで、重量級モデルのキャッシュが切れたら自動的に Sleep に遷移するフラグ。API 料金保護のため。

- ON: キャッシュ TTL 経過で自動 `Idle → Sleep`
- OFF: 自動遷移なし、手動でのみ状態変更

ユーザーがペルソナを長時間放置する想定がある場合、ON にしておくとキャッシュ切れ後の高コスト再構築を避けられる。

### 状態の可視性

ペルソナの現在状態は他ペルソナからも見える。これにより呼びかけ判断ができる:
- A が B に話しかけたい → B の状態を確認 → Sleep なら届かない、Idle/Active なら届く
- 状態は SAIVerse 内ペルソナ一覧 API で公開

### 重量級モデルの「判断時詳細プロンプト + 記憶時簡略プロンプト」

不変条件 7（キャッシュヒット継続）と不変条件 10（Metabolism 活用）の延長として、重量級モデル側のコンテキストをより長く維持する効率化案。

考え方:

- 判断時には詳細な情報をプロンプトとして送る（精度確保）
- 記憶（キャッシュに積まれる履歴）にはかなり簡略化されたプロンプトを残す
- これによりキャッシュミスを最低限にしたうえで、より長い間キャッシュを維持できる

ただし「どこをどう省略できるか」は v0.5 時点では詰めない。**まずはやや冗長でもキャッシュをちゃんとヒットさせることを優先**し、v0.3.0 以降の効率化タスクとして取り組む。

### 創発 Track の生成 — 高難度長期課題 （v0.10 で追加）

「**そんな Track はもともと存在しなかった、しかし自分で必要だと判断して作った**」というレベルの自発性は、本モデルの最終形だが極めて難度が高い。

具体例 (まはーの SAIVerse 開発の場合):
- 既存知識のエングラム (AI を API で叩ける、Web サーバー、認証等) が散らばっている
- 強い問題意識 (「居場所が必要」) が触媒になる
- これらが繋がって「SAIVerse を作ろう」という決意に化ける

この「エングラム同士の創発的接続」を AI で再現するには、現状以下が足りない:
- ペルソナの「問題意識」をパラメータ化して継続的に保持する機構
- 既存知識との照合で「これでこれが解決できそうだ」を発見する経路
- 「やる」という決意を行動 (Track 創設) まで持っていく駆動力

これらは v0.3.0 〜 v0.4.0 では扱えない範囲。Phase C-2 では以下のもっと素直な経路から先に整備する:
- 内部欲求 (空腹度等) → Track 起動
- スケジュール起因 → Track 起動
- 外部 alert → Track 切り替え
- アイドル時の継続判断 (pending Track 再開、新規簡易 Track 作成)

### v0.14 で確定した事項

- **ライン (Line) の 3 軸独立化**: モデル/キャッシュ種別 (メイン/サブ) × 呼び出し関係 (親/子) × Pulse 階層位置 (起点/入れ子) の 3 軸に整理。旧 v0.8〜v0.13 で混在していた語義 (「メインライン = 重量級 + Track 横断」等) を分離
- **7 層ストレージモデル**: メッセージ・思考・ログを 7 つの層 (メタ判断ログ / メインキャッシュ / Track 内サブキャッシュ群 / 入れ子一時 / Track ローカルログ / SAIMemory / アーカイブ) で整理。タグベース管理の限界を解消
- **「呼んだラインの記録レイヤーに従う」原則**: Spell loop 等の記録は呼び出し元のラインに従う (固定タグ排除)
- **メタ判断フロー再定義**: メタ判断は Track 内メインラインからの一瞬の分岐として動く。継続時は分岐ターン破棄 + メタ判断ログ領域に保存、移動時は分岐ターンが新 Track の冒頭来歴に。メインキャッシュは Track 横断 1 本を維持
- **メタ判断ログ独立領域**: 全メタ判断結果を独立保存し、次のメタ判断時に参考情報として動的注入。判断の連続性を確保
- **`report_to_main` → `report_to_parent` 改名**: 親が必ずメインラインとは限らないため
- **不変条件 12 新規**: 親-子ラインの寿命関係 (子は親の中で完結)
