# Intent Document: 動的状態同期（Dynamic State Sync）

## 概要

SAIVerse世界の動的な状態変化を、LLMのキャッシュヒット率を損なわずにペルソナへ通知するための設計。

---

## なぜ必要か（解決する問題）

### キャッシュヒット問題

LLMのプロンプトキャッシュは「コンテキストの先頭から一致している部分」に対して有効になる。現状の設計では以下の情報がコンテキスト上部（システムプロンプトや会話履歴の早い位置）に置かれており、変化のたびにキャッシュが全滅する：

- **ビジュアルコンテキスト**: アイテム追加・削除・入退室のたびにリアルタイム更新 → 以降の会話履歴すべてがキャッシュヒットしなくなる
- **ワーキングメモリ（`## 現在の状況`）**: `recalled_ids` や `situation_snapshot` などがシステムプロンプトに注入されており、変化のたびにシステムプロンプトが変動する
- **Memopedia**: 書き込み・更新がリアルタイムに反映されてしまう

### Metabolism（新陳代謝）との連携

SAIVerseには既にMetabolism機構（コンテキストウィンドウから古いメッセージを押し出す処理）が存在する。この機構をフル活用することで、「最初の1回のキャッシュミス後はMetabolismまでキャッシュヒットし続ける」設計が実現できる。

---

## 設計：A/B/Cの3状態モデル

SAIVerse世界の各状態について、以下の3つを管理する。

| 状態 | 意味 | 保存方法 |
|---|---|---|
| **A** | Metabolism直後にペルソナが認識していた状態（ベースライン） | DBに永続化 |
| **B** | 現在のコンテキストにイベントメッセージとして反映済みの状態 | B = A + Σ(コンテキスト内のイベントメッセージ) で導出。保存不要 |
| **C** | SAIVerse世界の現在の実際の状態 | 各モジュールがリアルタイムで保持 |

### 基本原則

- **C ≠ B のときのLLM呼び出し**: 差分をイベントメッセージとして会話履歴の末尾に挿入する
- **Metabolism時**: コンテキストから押し出される部分のイベントメッセージを積算してB'を構築し、AをB'に置き換える
- **コンテキスト最上部（A）は固定**: Metabolism後に1度だけ書き換えられる。パルスをまたいでは変化しない

---

## イベントメッセージ

動的な変化をペルソナに通知するために会話履歴に挿入するメッセージ。

### 特性

- 通常の会話メッセージと同じく、コンテキストウィンドウから自然に押し出される（揮発性）
- 押し出されるタイミングでMetabolismがAに反映するため、情報は失われない
- コンテキスト末尾（最新）付近に挿入されるため、以前のキャッシュを壊さない

### 想定フォーマット（例）

```
[システム通知]
- エア が「創造の祭壇」から退室しました
- アイテム「青いノート」(b:3) が追加されました
- Memopedia「天気のまとめ」が更新されました
```

複数の変化は1つのメッセージにまとめて挿入することでノイズを減らす。

---

## 検知スコープ

各ペルソナが検知すべき変化は「現在いるBuildingで知覚できる情報」に限定される。

| 情報の種類 | スコープ |
|---|---|
| 自分のインベントリ・Memopedia・Chronicle | 常に対象 |
| 自分自身の変動（名前・プロンプト・外見） | 常に対象 |
| 今いるBuilding内のアイテム・Buildingの変動 | 対象 |
| 今いるBuildingへの入退室（他ペルソナ・ユーザー） | 対象 |
| 同じBuildingにいる他者の公開情報（名前・外見） | 対象 |
| 同じBuildingにいる他者のシステムプロンプト | **対象外**（内部情報） |
| 別のBuildingの出来事・別Buildingにいる人の変動 | **対象外** |

---

## 対象カテゴリ

以下の状態変化がイベントメッセージの対象：

| カテゴリ | 変化の種類 |
|---|---|
| Building内アイテム | 追加・削除・名前変更・移動 |
| インベントリ内アイテム | 追加・削除・名前変更 |
| Memopedia | ページ作成・更新・削除 |
| Chronicle | エントリ追加（※Metabolismと同期済みのため軽微な対応で済む可能性あり） |
| 他ペルソナの入退室 | 入室・退室 |
| ユーザーの入退室 | 入室・退室 |
| ペルソナ自身のBuilding移動 | 後述 |
| Buildingの変動 | 内装変化・名前変更・システムプロンプト変更 |
| ペルソナ自身の変動 | 名前変更・システムプロンプト変更・外見画像変更 |
| 他ペルソナの変動 | 名前変更・外見画像変更（同じBuildingにいる間のみ） |
| ユーザーの変動 | 名前変更（同じBuildingにいる間のみ） |
| ピン止めメッセージ | 追加・削除 |

## Building移動の特殊処理

ペルソナ自身がBuildingを移動する場合、通常のイベントメッセージとは異なる扱いが必要になる。

### 基本方針：Aはリセットしない

移動のたびにAをリセットしてコンテキスト最上部を再構築すると、移動のたびにキャッシュが全滅する。これを避けるため、**Aは直近のMetabolism時の状態のまま維持し、移動先の情報はイベントメッセージとして会話履歴末尾に挿入する**。

### 到着イベントメッセージ

移動時に挿入する特殊なイベントメッセージ。移動先Buildingの現在状態を「その場を見回して得られた情報」として読み込む。

- **初訪問**: 移動先の現在状態を丸ごとスナップショットとして挿入
- **再訪問**: 前回退室時の認識からの差分のみを挿入（「あれ、新しいアイテムを作ったの？」という自然な発話が可能になる）

### Building別・最終認識状態（last_known_state）

ペルソナごとに、訪問したことがあるBuildingの「最後に知覚した状態」をDBに保持する。

| タイミング | 処理 |
|---|---|
| 初訪問 | 現在状態をlast_known_stateとして記録 |
| 滞在中のMetabolism | 現在BuildingのA昇格と同期してlast_known_stateを更新 |
| 退室時 | last_known_stateをその時点で**凍結**。退室後の変化は追跡しない |
| 再訪問 | 凍結されたlast_known_state → 現在状態の差分のみを到着イベントメッセージとして挿入 |

退室後に訪問先で何かが変化していても、ペルソナは再訪するまで知らない。これは「帰宅中に何かあった」を到着時に初めて知るという自然な挙動に対応している。

## Metabolismとの統合

Metabolism処理に以下のフックを追加する：

1. **押し出されるメッセージの特定**: コンテキストから除外されるメッセージ群の中にイベントメッセージが含まれるかチェック
2. **B'の構築**: A + 押し出されるイベントメッセージによる変化差分 = B'
3. **AをB'に昇格**: B'をDBに保存し、次のMetabolism時の新しいAとする
4. **コンテキスト最上部の再構築**: 新しいAに基づいてビジュアルコンテキスト等の「初期状態表現」を再生成する

---

## ワーキングメモリの再設計

この設計により、既存のワーキングメモリの役割が以下のように整理される：

| 旧フィールド | 新しい扱い |
|---|---|
| `recalled_ids` | 廃止。想起時に内容を直接会話履歴に挿入する。システムプロンプトへの注入も廃止 |
| `situation_snapshot` | 廃止。A/B/Cモデルに吸収（A が旧snapshotに相当） |
| カスタムキー（`update_working_memory`） | ピン止めメッセージとして再定義。追加・削除はイベントメッセージで通知 |

ワーキングメモリをシステムプロンプトに注入する `## 現在の状況` セクションは廃止する。

---

## 不変条件

- **コンテキスト上部を変更しない**: Metabolism以外のタイミングでビジュアルコンテキスト・Aの内容を書き換えてはならない
- **イベントメッセージは末尾に追加するのみ**: 既存のメッセージを遡って変更しない
- **B = A + Σが常に成立するように管理する**: イベントメッセージの挿入と飛ばしが発生しないよう、差分検出は確実に行う
- **同一変化の重複通知禁止**: C = B の場合はイベントメッセージを挿入しない

---

## 実装方針

### フェーズ1：共通インフラ

1. イベントメッセージの挿入インターフェース定義
2. A状態の永続化スキーマ設計（カテゴリごとのスナップショット形式）
3. Metabolismフックの追加（B'構築 → A昇格）
4. C ≠ B の差分検出の共通ロジック

### フェーズ2：カテゴリ別実装

各カテゴリを独立して実装・テスト可能。優先度は影響の大きいものから：
1. Building内アイテム・インベントリ（ビジュアルコンテキスト更新停止と連動）
2. 入退室（`situation_snapshot` の廃止と連動）
3. Memopedia
4. その他

### フェーズ3：ワーキングメモリ整理

- `recalled_ids` 関連コードの削除
- `situation_snapshot` 関連コードの削除
- `## 現在の状況` システムプロンプト注入の削除
